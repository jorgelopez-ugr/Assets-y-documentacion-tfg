MVP 3.5 — Design Changes

⚠ Documento histórico (pre-refactor)

Refleja un estado anterior al refactor de caminos forestales (jun 2026); se conserva como traza del proceso. El estado actual está en Inicio.

← Inicio · Arquitectura del Sistema · Arquitectura API

Por qué este MVP existe

El sistema anterior asumía que el terreno puede cambiar entre sesiones. No es así. El DEM (heightmap) y la imagen hiperespectral son datos fijos, conocidos y nunca cambian. No tiene sentido que Godot calcule y envíe 100.000 celdas por la red cada vez que arranca. El estado inicial es determinista y debe precalcularse una sola vez offline.

Arquitectura objetivo

[OFFLINE — se ejecuta una vez, resultado versionado]
heightmap_16bit.png
        ↓
tools/build_initial_state.py
        ↓
data/initial_state.json   ← fuente de verdad del sistema

[RUNTIME]
API arranca → carga data/initial_state.json automáticamente
Godot → POST /ignite {"x": N, "z": N}  → enciende una celda
Godot → POST /tick                      → avanza la simulación
Godot → GET /active_cells               → recibe celdas activas
Godot → POST /reset                     → vuelve al estado inicial

Issues implementadas

#24 — tools/build_initial_state.py

Script offline que sustituye a terrain_to_json.py. Genera data/initial_state.json con: - 100×100 celdas de superficie, cada una con fuel_coefficient: 1.0 - Estado inicial EMPTY para todas las celdas - Bloque params: k=2.0, dt=0.5, A=0.1, B=0.5, iter=5

⚠ Outdated

Tests: 10/10

#25 — Motor C++: fuel_coefficient

  • engine/celda.h — nuevo campo float fuel_coefficient = 1.0f
  • engine/reaccion.h — reaction -= k * dt * fuel_coefficient
  • engine/serializar.h — lee y escribe fuel_coefficient (default 1.0 si falta)

Con fuel_coefficient = 0.0 una celda nunca arde — base para el futuro mapa hiperespectral de vegetación.

Tests: 8/8

#26 — API: arranque automático y nuevos endpoints

  • lifespan de FastAPI carga data/initial_state.json al arrancar
  • POST /reset — sin body, sin modelo Pydantic, recarga el fichero
  • POST /ignite {"x": N, "z": N} — enciende la celda de superficie de esa columna
  • motor.configurar_params() — relanza el proceso C++ si los params del fichero cambian
  • state.cargar_desde_fichero() — carga estado + construye índice de superficie para /ignite

Ver: Arquitectura API

Tests: 31/31

#27 — Godot: adaptar simulacion.gd (pendiente — responsabilidad del usuario)

Eliminar _cargar_terreno() y la lógica de POST /reset con body. Añadir POST /ignite al hacer click en el terreno con raycast. Ver issue #27 en GitHub para los detalles técnicos.

Criterio de cierre

  1. data/initial_state.json versionado, con fuel_coefficient y bloque params
  2. La API arranca sola — Godot no necesita hacer ninguna llamada inicial
  3. POST /reset sin body restaura el estado inicial
  4. POST /ignite enciende una celda correctamente
  5. El motor respeta fuel_coefficient: 0.0
  6. Todos los tests pasan

Segunda iteración — Sistema de logging y debug mode

Añade observabilidad completa sin cambiar la lógica de simulación.

#29 — api/logger.py + logs/

Módulo centralizado de logging con RotatingFileHandler (5 MB, 3 backups) y handler de consola opcional. Controlado por la variable de entorno DEBUG_MODE=1.

Ver: Sistema de Logging

#30 — Debug mode en motor C++

Flag --debug en el motor: escribe stats por stderr tras cada tick (celdas BURNING/SMOKE, temperatura máxima, tick count). La API captura el stderr y lo escribe en logs/engine.log.

Ver: Sistema de Logging

Tests: 3/3

#31 — logs/rotate_logs.py

Script de rotación manual. Comprime los .log que superan los umbrales (1 MB o 7 días) como .log.YYYY-MM-DD.gz. Vacía el original sin eliminarlo para no romper los handlers activos.

Ver: rotate_logs

Tests: 6/6

Lo que NO entró en este MVP

  • Integración de la imagen hiperespectral (fuel_coefficient real por especie)
  • Cámara libre tipo videojuego en Godot
  • Optimización sparse del solver y advección
  • Shaders de fuego
  • Caminos forestales
