build_world.py — leyendo las capas hiperespectrales de especies

← Inicio · Integracion Godot-Motor — Recopilatorio · int16-overflow-build-world

Milestone: Integración Godot-Motor (Fase 2 #81/#82) Archivos: tools/build_world.py · docs/references/color-*.png

Contexto

Hasta este milestone, tools/build_world.py generaba data/world.dat con fuel_coefficient = 1.0 uniforme para todas las celdas dentro del DEM. La Fase 2 cambia eso: el fuel_coefficient ahora se deriva de las 7 capas hiperespectrales que el usuario tiene escaneadas del terreno real (Puerto La Mata).

Las 7 capas de entrada

En docs/references/ hay siete PNG, todas de 1058×539 px, cada una con la ortofoto del terreno escaneado más una capa pintada con un color específico para una categoría / especie:

Archivo Color RGB Categoría fuel asignado
color-vegetacion-alta.png (0, 0, 255) Vegetación alta agregada 0.90
color-vegetacion-baja.png (255, 0, 0) Matorral / sotobosque 0.55
color-pinus-helepensis.png (0, 255, 255) Pinus halepensis (pirofítico) 0.95
color-encina.png (127, 0, 255) Quercus ilex (resistente fuego) 0.40
color-quejigo.png (255, 128, 0) Quercus faginea (caducifolio) 0.50
color-terreno.png (0, 255, 0) Suelo desnudo 0.00
color-camino.png (255, 0, 255) Camino (cortafuegos natural) 0.00

Los colores se obtuvieron empíricamente con un script que cuenta los píxeles más saturados de cada PNG. Verificado contra la clasificación visible en etiquetas-especies.png.

Especies "invisibles" del catálogo

Hay otras especies del catálogo docs/references/Caracteristicas de especies.txt (Berberis hispanica, Echinospartum boissieri, Erinacea anthyllis, Stipa tenacissima) cuyo color en la clasificación hiperespectral es despreciable (< 100 píxeles, lost in noise). El usuario aceptó dejarlas fuera del mapeo:

"Hay muchas especies que la verdad que el color es invisible yo supongo que porque desde el espacio es irrisorio, las que son demasiado poco que ni se ven no te preocupes porque no es culpa nuestra."

El criterio fuel por especie

Sin tabla numérica en el catálogo, los valores asignados se derivan de:

  • Pinus halepensis (0.95): la propuesta del TFG lo identifica como especie pirofítica, "favorece incendios". Resinoso, combustible alto.
  • Vegetación alta agregada (0.90): categoría compuesta (pinos, encinas densas, otras). Valor alto razonable como "hay biomasa".
  • Vegetación baja / matorral (0.55): menor densidad de biomasa que la alta, pero combustible significativo (encina rastrera, romero, brezo).
  • Quejigo (0.50): caducifolio, combustibilidad media.
  • Encina (0.40): perenne, hojas duras, resistente al fuego. Citada en la propuesta como contrapunto biológico de los pinos.
  • Suelo desnudo (0.00) y camino (0.00): nada que arda; actúan como cortafuegos naturales.

Si en una iteración posterior se obtiene una tabla numérica de combustibilidad por especie de la literatura, basta con cambiar los valores en el dict MAPEO_ESPECIES en build_world.py.

Algoritmo de combinación

MAPEO_ESPECIES: list[tuple[str, tuple[int,int,int], float]] = [
    ("color-terreno.png",          (0,   255, 0  ), 0.00),
    ("color-vegetacion-alta.png",  (0,   0,   255), 0.90),
    ("color-vegetacion-baja.png",  (255, 0,   0  ), 0.55),
    ("color-pinus-helepensis.png", (0,   255, 255), 0.95),
    ("color-encina.png",           (127, 0,   255), 0.40),
    ("color-quejigo.png",          (255, 128, 0  ), 0.50),
    ("color-camino.png",           (255, 0,   255), 0.00),  # último = máxima precedencia
]
COLOR_TOLERANCIA = 10
FUEL_DEFAULT = 0.0

def cargar_fuel_map(width, depth):
    fuel = np.full((depth, width), FUEL_DEFAULT, dtype=np.float32)
    for archivo, color_esperado, valor_fuel in MAPEO_ESPECIES:
        img = Image.open(REFERENCES / archivo).convert("RGB")
        if img.size != (width, depth):
            img = img.resize((width, depth), Image.NEAREST)
        pixels = np.asarray(img, dtype=np.int32)   # ¡int32!
        diff = pixels - np.array(color_esperado, dtype=np.int32)
        dist2 = diff[..., 0]**2 + diff[..., 1]**2 + diff[..., 2]**2
        mascara = dist2 <= COLOR_TOLERANCIA**2
        fuel[mascara] = valor_fuel
    return fuel

Orden de aplicación con precedencia

Las capas se procesan en orden y la última gana en píxeles solapados. Esto sirve para distinguir:

  • Agregada antes que específica: vegetación-alta (azul) cubre toda la zona de bosque. Luego pinus, encina y quejigo se aplican encima de píxeles que ya están como vegetación-alta y los sobreescriben con su valor específico.
  • Camino al final: máxima precedencia, porque un camino que cruza una zona de pinus debe verse como camino (fuel 0).

Matching por distancia euclídea RGB con tolerancia 10

Es necesario para absorber el antialiasing de la clasificación hiperespectral (píxeles (1, 1, 253) siguen siendo "azul"). 10 unidades es suficiente para esto y demasiado bajo para colisionar con tonos naturales de la ortofoto (verdes de árbol real, marrones de tierra).

Alineación con homografía SIFT (#139)

✎ Actualizado en #139

Antes se estiraba la imagen hiperespectral al DEM sin preservar ratio (1058×539 → 1598×1393), una distorsión asumida que descuadraba la textura. Ahora se alinea con visión por computador: tools/alinear_ hiperespectral.py detecta la silueta del escaneo y estima una homografía (ORB/SIFT + RANSAC) entre las ortofotos para cuadrar las capturas sobre el DEM. build_world.py consume esa textura alineada (no el estirado naive) para derivar el fuel. Ver el issue #139.

La fidelidad geográfica plena seguiría necesitando un GeoTIFF georeferenciado (CRS EPSG:32630), pero la homografía cuadra las capturas mucho mejor que el estirado.

[FUEL] resumen: 47.4% fuel=0 (no arde), 51.2% fuel≥0.8

Lectura:

  • 47.4 % no arde (suelo desnudo + caminos + no clasificado).
  • 51.2 % arde rápido (vegetación alta + Pinus).
  • ~1.5 % arde con menor velocidad (matorral + quejigo + encina).

Coherente con la imagen visible: el centro del mapa es bosque denso clasificado (azul mayoritario, con manchas naranjas de quejigo) y los bordes son terreno expuesto.

Impacto en tests

Los tests asumían fuel=1.0 uniforme en el cuadrante (5, 5) — coords (275, 275). Con Fase 2 esa celda tiene fuel=0. Tres acciones:

  1. test_world_loader.cpp: sustituido el test "todo es 1.0" por fuel_en_rango, fuel_no_uniforme y fuel_valores_canon (este último verifica que cada celda pertenece al conjunto {0.0, 0.4, 0.5, 0.55, 0.9, 0.95}).

  2. test_simulacion.cpp + test_e2e.py: coordenadas migradas al cuadrante (16, 2) — coords (825, 125) — donde hay una zona de vegetación alta densa de 100×100 celdas con fuel uniforme 0.90, lo que permite propagación lateral como en el snapshot anterior.

  3. test_simulacion.cpp::test_determinismo_numerico_adveccion: snapshot recalculado para el cuadrante (16, 2). La simulación sigue siendo bit-determinista; los nuevos valores son la nueva referencia.

Bug significativo del camino

Ver int16-overflow-build-world. El primer intento usaba dtype=np.int16 por costumbre de imágenes; los cuadrados overflowaban y la máscara matcheaba ~99 % del mapa. Cambio a int32. Lección: para imágenes con NumPy, int32 o float32 por defecto.
