Error: buoyancy aplicada a celdas FUEL destruye su temperatura antes de que reacción pueda actuar

← Inicio · Pipeline de Simulación · MVP 4 — Propagación real de fuego

MVP afectado: MVP 4 Archivos involucrados: engine/buoyancy.h · engine/adveccion.h · engine/reaccion.h

El síntoma

Al ejecutar el test de integración #9 — "celda FUEL a 200°C pasa a BURNING" — el motor devolvía la celda en estado FUEL en lugar de BURNING, a pesar de que el umbral de ignición es 100°C y la celda partía de 200°C.

Lo llamativo: el test unitario test_fuel_caliente_pasa_a_burning (en test_reaccion.cpp) sí pasaba. La función aplicarReaccion por sí sola funcionaba correctamente. El fallo solo aparecía al pasar el JSON por el motor completo.

[OK]  test_fuel_caliente_pasa_a_burning   ← test unitario pasa
[FAIL] Celda FUEL a 200°C no pasó a BURNING  ← integración falla

Cómo se encontró

El test unitario prueba aplicarReaccion de forma aislada — le entrega una celda con temperatura=200 y verifica que pase a BURNING. Correcto.

El test de integración pasa el mismo JSON al binario motor, que ejecuta el pipeline completo:

aplicarBuoyancy  →  resolverPresion  →  advectar  →  aplicarReaccion

La diferencia entre ambos tests era exactamente el pipeline previo a aplicarReaccion. Eso acotaba el problema a buoyancy, presion o adveccion.

La causa raíz

aplicarBuoyancy en engine/buoyancy.h iteraba sobre todas las celdas del grid sin filtrar por estado:

// CÓDIGO INCORRECTO (antes del fix)
void aplicarBuoyancy(Grid& g, float A, float B) {
    for (...) {
        Celda& c = g.get(x, y, z);
        float fuerza = A * c.temperature - B * c.density;
        c.velocity.z += fuerza;   // ← aplica a CUALQUIER celda
    }
}

Para una celda FUEL con temperatura=200°C y parámetros por defecto (A=0.1, B=0.5, density=0):

fuerza = 0.1 * 200 - 0.5 * 0 = 20.0
velocity.z = 0 + 20 = 20.0

Luego en la advección MacCormack, el método semi-Lagrangiano traza hacia atrás:

origen_z = z - dt * velocity.z
         = 1 - 0.1 * 20
         = 1 - 2
         = -1   ← fuera del grid, se clampea a z=0

La posición de origen clampea a una celda EMPTY con temperatura=0°C. La interpolación trilineal devuelve 0°C para la temperatura de la celda FUEL. Cuando llega aplicarReaccion:

temperature = 0°C  <  umbral 100°C  →  FUEL permanece FUEL

El calor "desapareció" antes de llegar a la etapa que decide si la celda enciende.

La solución

Restringir aplicarBuoyancy a las celdas que realmente son gas: BURNING y SMOKE. Las celdas FUEL son combustible sólido — no flotan ni tienen dinámica de fluido.

// engine/buoyancy.h — después del fix
void aplicarBuoyancy(Grid& g, float A, float B) {
    for (...) {
        Celda& c = g.get(x, y, z);

        // Buoyancy solo aplica a gases (fuego y humo), no a combustible sólido
        if (c.state != EstadoCelda::BURNING && c.state != EstadoCelda::SMOKE) continue;

        float fuerza = A * c.temperature - B * c.density;
        c.velocity.z += fuerza;
    }
}

Con este cambio, una celda FUEL conserva velocity=(0,0,0). La advección traza hacia atrás a su propia posición y recupera su temperatura original. aplicarReaccion la recibe a 200°C > 100°C y la transiciona a BURNING.

Por qué tiene sentido físicamente

La fuerza de flotabilidad modela el ascenso del gas caliente (columna de aire caliente, humo). Es una propiedad de los fluidos, no de los sólidos.

Una celda FUEL representa vegetación: hierba, matorral, árbol. No flota. No tiene velocidad propia. Solo recibe calor de sus vecinas ardiendo y, cuando ese calor supera el umbral de ignición, enciende.

Aplicar buoyancy a FUEL no era solo un bug de implementación — era un error conceptual en el modelo físico.

Señal que delató el problema

La divergencia entre test unitario y test de integración fue la clave. Cuando una función pasa sus tests unitarios pero falla en integración, el problema está casi siempre en la interacción entre módulos, no en la función en sí. En este caso, el pipeline previo alteraba los datos antes de que llegaran al módulo que fallaba.

Tests añadidos tras el fix

En engine/test_buoyancy.cpp: - test_buoyancy_no_afecta_fuel — una celda FUEL con 500°C no modifica su velocidad - test_buoyancy_no_afecta_empty — una celda EMPTY tampoco recibe buoyancy

Tests previos actualizados para usar estado BURNING o SMOKE explícitamente, dejando claro que buoyancy requiere gas activo.
