Cortafuegos — `#70` — `FIREBREAK_UNDO_BATCH` y restore en `/reset`

⚠ Documento histórico (pre-refactor)

Refleja un estado anterior al refactor de caminos forestales (jun 2026); se conserva como traza del proceso. El estado actual está en Inicio.

← Cortafuegos — 69 — Comando FIREBREAK_BATCH · MVP — Arquitectura de Cuadrantes · Cuadrante y Celda

TL;DR

  • Segunda pieza del milestone "Cortafuegos pintables desde Godot": el motor expone un comando FIREBREAK_UNDO_BATCH que deshace cortafuegos previamente pintados y modifica Simulacion::reset() para restaurar los fuel_coefficient desde el mapa de miguitas introducido en #69.
  • /reset ya no necesita releer world.dat: con N ≤ 1000 celdas pintadas en una sesión típica, el coste es O(N) sobre el mapa de miguitas. Mucho más barato que un I/O al disco.
  • Determinismo end-to-end: pintar + reset + ignite + 5 ticks produce el mismo estado celda-por-celda que ignite + 5 ticks desde un mundo recién arrancado. Garantía que no teníamos sin restore.
  • Nivel B en el undo: si el cuadrante está ACTIVO y la Celda viva está en AIR con fuel original > 0, el undo la vuelve a FUEL. Si está en FUEL (pincel parcial deshecho), restaura cell.fuel_coefficient. Si está BURNING/SMOKE, NO se toca (decisión 1: no des-apagar).
  • Cero regresiones: 31 motor (test_main.sh) + 74 motor (test_simulacion.sh) + 46 API. 24 asserts nuevos.

Por qué hace falta esta issue

Sin el restore, una sesión de Godot pintaría cortafuegos, simularía, luego pulsaría "reset" para empezar de nuevo… y el mapa quedaría permanentemente alterado. La única forma de volver a un estado limpio sería matar el motor y relanzarlo (que sí relee world.dat). UX inaceptable.

#69 dejó el mapa de miguitas listo para esto. #70 cierra el ciclo.

Decisiones cerradas (HANDOFF, decisión 5)

Aspecto Decisión
DELETE de celda no pintada No-op silencioso (no error).
DELETE en cuadrante activo + AIR + original > 0 Volver a FUEL con el original.
DELETE en cuadrante activo + FUEL (parcial deshecho) Restaurar cell.fuel_coefficient.
DELETE en BURNING/SMOKE NO tocar la Celda viva. La estática SÍ se restaura.
/reset Iterar miguitas, restaurar todo, limpiar mapa. NO releer disco.

La decisión sobre BURNING/SMOKE merece un matiz: si una celda fue pintada (fuel_coefficient estática puesta a 0) y luego, por convección, reignitó como BURNING, deshacer el cortafuegos: - restaura la copia estática al original (la próxima vez que se reactive el cuadrante con activar, leerá el valor restaurado); - pero NO toca la Celda viva, que sigue ardiendo con su dinámica.

Es coherente: la motosierra no apaga incendios, así que tampoco "des-apaga".

Diseño

simulacion.h::deshacer_firebreak

void deshacer_firebreak(int wx, int wz);

Algoritmo (espejo de pintar_firebreak):

  1. Validación de wx<0 || wz<0 y rango de cuadrantes.
  2. Consulta: key = (qid << 32) | idx. Si no está en fuel_modificado_, no-op (silencioso).
  3. Restore estático: c.fuel_coefficient[idx] = original. Borrar entrada del mapa.
  4. Nivel B: si el cuadrante está ACTIVO: - state == FUEL → cell.fuel_coefficient = original. - state == AIR && original > 0 → state = FUEL, cell.fuel_coefficient = original. - BURNING || SMOKE → no tocar la viva.

simulacion.h::reset

Antes de desactivar cuadrantes (limpiar SubGrids, vaciar activos_), iteramos el mapa de miguitas y restauramos cada fuel_coefficient a su valor original:

for (const auto& [key, original] : fuel_modificado_) {
    const uint32_t qid = static_cast<uint32_t>(key >> 32);
    const int idx = static_cast<int>(key & 0xFFFFFFFFu);
    world_.cuadrantes[qid].fuel_coefficient[idx] = original;
}
fuel_modificado_.clear();

Por qué basta con la copia estática y no necesito tocar SubGrids: los SubGrids se vacían justo después (c.grid.cells.clear()). Cuando el cuadrante vuelva a activar en una sesión futura, leerá el fuel_coefficient restaurado y reconstruirá la Celda con state = FUEL natural.

main.cpp::FIREBREAK_UNDO_BATCH

Cabecera:

FIREBREAK_UNDO_BATCH <N>
<x1> <z1>
<x2> <z2>
...
<xN> <zN>

Respuesta:

FIREBREAK_UNDO_DONE <N_procesadas>

A diferencia de FIREBREAK_BATCH, no hay intensidad — el restore consulta el original guardado.

Coste de /reset con N celdas pintadas

/reset añade un loop O(N) sobre el mapa de miguitas. Estimaciones:

  • N = 100 (sesión corta, una franja pequeña): ~unos pocos µs.
  • N = 1000 (varias franjas, caso típico): ~50 µs.
  • N = 10000 (caso patológico, brocha barriendo medio mapa): ~500 µs.

Comparado con la lógica de reset previa (vaciar SubGrids de todos los cuadrantes activos), el coste de la restauración de miguitas es despreciable. No vale la pena medirlo en bench.

Decisiones tomadas durante la implementación

  1. Restore estático antes de tocar SubGrids. La alternativa era restaurar también la Celda viva en cada cuadrante activo. Pero los SubGrids se vacían en el mismo reset(), así que no aporta nada — la próxima activar() los reconstruirá desde la estática ya restaurada. Una sola fuente de verdad.

  2. No relelectura de world.dat. El mapa de miguitas es la memoria de "qué se cambió" en esta sesión. Releer el disco sería I/O innecesario y rompería la propiedad de que el motor es un proceso persistente independiente del filesystem.

  3. deshacer_firebreak valida wx<0 || wz<0. Mismo razonamiento que en #69: división entera C++ trunca hacia cero, así que el filtro de cuadrante no detecta coordenadas negativas. Necesario para no acceder a fuel_coefficient[-1].

  4. Nivel B en undo restaura también el caso FUEL parcial. Si se pintó con intensidad=0.5, la celda sigue siendo FUEL pero con fuel reducido. Al deshacer, hay que restaurar cell.fuel_coefficient aunque el state no cambie. Es lo que hace if (state == FUEL).

  5. Test de determinismo end-to-end. Vale más que cinco asserts sueltos: pintar + reset + ignite + 5 ticks debe ser bit-equivalente a ignite + 5 ticks limpio. Si hay alguna fuga de estado (algún campo del SubGrid que no se vacía bien tras reset, alguna miguita no restaurada), este test lo caza.
