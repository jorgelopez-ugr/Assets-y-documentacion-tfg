TL;DR

  • Cuarta y última pieza del milestone "Viento" (la tercera issue cronológicamente). Tests E2E HTTP + script de validación narrativa que demuestran empíricamente que el viento desplaza el frente.
  • Hipótesis cumplida en ambos ejes con magnitud 10 m/s a 30 ticks: Δcx = +17.4 con viento (10, 0); Δcz = +11.3 con viento (0, 10).
  • El test crítico de la convención .y → Z mundo PASA end-to-end: set_viento(0, 10) empuja el centroide en cell.z, no en cell.y (vertical). Confirma que el swap en Simulacion::set_viento está bien implementado y que la API es una superficie limpia.
  • 3 tests E2E nuevos + script api/viento_validacion.py. Cero regresiones.

Por qué esta issue cierra el milestone

Las dos issues previas dejaron motor + API operativos. #76 verifica que el sistema completo, integrado, demuestra la hipótesis física que justifica todo el milestone (el viento desplaza el frente) y que la trampa de los ejes está bien resuelta a lo largo de toda la cadena motor ↔ API ↔ tests.

Si los tests pasan en X pero fallaran en Z, sería bug en el swap de set_viento (Vec3.y ↔ Z mundo). Ese diagnóstico está explícito en el assert del test crítico para facilitar el debug.

Decisiones tomadas durante la implementación

  1. Centroide como métrica primaria. La alternativa habría sido "número de celdas BURNING en un sector del mapa" (como hicimos con cortafuegos y el "corredor"). Pero en viento el efecto es un desplazamiento, no una contención: la métrica natural es el centro de masa del frente, no un conteo en una región fija.

  2. Diferencia vs baseline, no posición absoluta. El centroide sin viento no está exactamente en (275, 275) por asimetría del solver y del DEM (pendiente del terreno). Esa asimetría se cancela al restar el baseline, dejando sólo la contribución del viento.

  3. Tests E2E a 25 ticks, no 30. A 25 ticks los márgenes ya son ≥ 2× los umbrales — más que suficiente para robustez. A 30 ticks los tests duplicarían su tiempo sin aportar más fiabilidad.

  4. Umbrales asimétricos (5 para X, 3 para Z). La asimetría base del solver favorece X levemente, así que viento en X se nota más. Los umbrales reflejan eso. No se trata de hacer el test "fácil": se trata de que el umbral sea coherente con la magnitud típica del ruido en cada dirección.

  5. Assert del test crítico con hint de diagnóstico. El mensaje de error del test_e2e_viento_Z_desplaza_centroide_en_cell_z incluye "Posible bug en el swap Vec3.y ↔ Z mundo (ver docs/convenciones-ejes.md)". Si alguien rompe el swap en un cambio futuro, el debug arranca con la pista correcta.

  6. test_e2e_reset_borra_viento toca también la API. Aunque ya hay un test C++ (test_reset_borra_viento en test_simulacion.cpp), este test E2E cierra el ciclo HTTP → motor → reset → motor. Es la prueba de que el cliente Godot puede pulsar "reset" y empezar limpio sin tener que llamar a POST /viento {0, 0} explícitamente.

  7. Bench narrativo a 30 ticks, tests a 25. Documentamos el bench con la versión más nítida (más ticks, más asimetría); los tests automatizados usan la versión más rápida con márgenes amplios.

Lectura para el TFG (gestores forestales)

Con 10 m/s de viento sostenido durante 30 ticks, el centro del incendio se desplaza 17 metros en la dirección del viento. En 5 minutos simulados, el frente ya no está donde lo dejaste — el viento manda. Un cortafuegos perpendicular al viento dominante es la herramienta que combina las dos features del simulador: usa la dirección del viento para decidir dónde cortar.

Esto enlaza con Cortafuegos — Recopilatorio: ahora una franja estrecha perpendicular al viento es eficaz, mientras que la misma franja paralela al viento se ve sobrevolada. Esa demostración combinada queda para un milestone futuro de "interacción cortafuegos × viento" si el TFG lo requiere.
