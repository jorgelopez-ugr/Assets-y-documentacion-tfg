TL;DR

  • Primera pieza del milestone "Cortafuegos pintables desde Godot": el motor expone un comando FIREBREAK_BATCH que recibe una intensidad y una lista de celdas y multiplica fuel_coefficient *= (1 - intensidad) en cada una.
  • No toca el bucle de tick: la latencia por tick no cambia. El comando se invoca a demanda desde la API (issue #71) cuando el usuario pinta en Godot.
  • Miguitas de pan: cada celda tocada por primera vez guarda su fuel original en un unordered_map<uint64_t, float> fuel_modificado_. Permitirá DELETE (#70) y restauración en /reset (#70).
  • Nivel B (decisión 4 del HANDOFF): si el cuadrante está ACTIVO y la Celda viva está en estado FUEL, el cambio se propaga a cell.fuel_coefficient y, si el nuevo valor es 0, la Celda pasa a AIR. Celdas BURNING/SMOKE NO se tocan (decisión 1: una motosierra no apaga incendios).
  • Cero regresiones: 27 motor (test_main.sh) + 54 motor (test_simulacion.sh)
  • 46 API. 12 asserts nuevos para esta issue.
  • Snapshot de determinismo: pasa sin tocar — la ruta de FIREBREAK no altera el cómputo físico cuando no se invoca.

miguitas y el handler del protocolo. Las tres siguientes issues dependen de estas estructuras.

Hipótesis física que justifica todo el milestone

Si el cortafuegos es demasiado fino y está rodeado de árboles con mucho fuel, la propagación del calor por convección puede hacer que el incendio salte igualmente. Crear correctos cortafuegos no es trivial — el ancho importa.

La narrativa la cierra la issue #72 con benches de franja estrecha vs ancha. Pero la mecánica ya sale gratis del solver:

  1. pintar_firebreak con intensidad=1 convierte Celda::state de FUEL a AIR, manteniendo el resto del SubGrid intacto.
  2. aplicarBuoyancy() empuja calor hacia arriba en celdas BURNING/SMOKE.
  3. advectar() lo transporta lateralmente por las celdas AIR (las pintadas).
  4. Si la franja es estrecha, la temperatura al otro lado puede superar UMBRAL_ACTIVACION = 100°C y reignitar.

No se ha tocado el solver. Sólo se le da al usuario el botón para modificar el campo fuel_coefficient desde fuera.

Decisiones cerradas (no re-abrir)

Recogidas en el HANDOFF y aplicadas literalmente en el código:

# Decisión Reflejo en el código
1 Pincel = FUEL → AIR. NO tocar BURNING/SMOKE. if (sup.state != EstadoCelda::FUEL) return;
2 Granularidad: intensidad ∈ [0,1], nuevo = actual * (1-intensidad). c.fuel_coefficient[idx] *= (1 - intensidad)
4 Nivel B: motor SIEMPRE acepta el comando, esté el cuadrante activo o no. El método procede sin importar c.estado.
5 Miguitas: la primera modificación guarda el original; repintar NO sobrescribe. if (fuel_modificado_.find(key) == end()) emplace(...)
6 Formato del comando: batch, no una línea por celda. FIREBREAK_BATCH <intensidad> <N> + N líneas <x> <z>

Decisión 6 se cerró durante la planificación tras pregunta explícita: una brocha en Godot genera lotes de cientos de celdas por trazo; un comando por celda satura stdin y multiplica el overhead de parseo.

Diseño

simulacion.h::pintar_firebreak

void pintar_firebreak(int wx, int wz, float intensidad);

Coordenadas mundiales enteras (índices de celda, una celda = 1 m). La API valida y pasa enteros; el motor no acepta floats — coherente con el payload JSON [[x,z], ...] que envía Godot.

Algoritmo:

  1. Validación: wx<0 || wz<0 → no-op. División entera C++ trunca hacia cero, así que wx=-1 daría qx=0 (no detectable con qx<0). Validamos directamente.
  2. Localización: qx = wx / CELDAS_QUAD_X, cx = wx - qx * CELDAS_QUAD_X. qid = qz * cuadrantes_x + qx.
  3. Miguita: key = (qid << 32) | idx. Si no está en fuel_modificado_, se inserta con el fuel original. Repintar no sobrescribe (decisión 5).
  4. Modificación estática: c.fuel_coefficient[idx] *= (1 - clamp(intensidad, 0, 1)).
  5. Nivel B (decisión 4): si c.estado == ACTIVO y la Celda viva está en FUEL: - nuevo_fuel <= 0 → state = AIR, cell.fuel_coefficient = 0. - nuevo_fuel > 0 → cell.fuel_coefficient = nuevo_fuel. - BURNING/SMOKE: la celda viva NO se toca.

El clamp de intensidad es paranoico (la API ya valida) pero protege contra inputs manuales por stdin — coste despreciable.

simulacion.h::fuel_modificado_

std::unordered_map<uint64_t, float> fuel_modificado_;
  • key: key_firebreak(qid, idx) = (qid << 32) | idx. idx máximo CELDAS_SUPERFICIE (2500), pero usamos 32 bits por simplicidad. Función pública estática para que tests y la issue #70 puedan recomponer claves.
  • value: fuel ORIGINAL antes de la primera modificación. La issue #70 lo consume para restaurar en DELETE / /reset.

Coste: O(N) en memoria, donde N = celdas pintadas en la sesión. Esperado N ≤ 1000 en uso normal.

main.cpp::FIREBREAK_BATCH

Cabecera:

FIREBREAK_BATCH <intensidad> <N>
<x1> <z1>
<x2> <z2>
...
<xN> <zN>

Respuesta:

FIREBREAK_DONE <N_procesadas>

N_procesadas puede ser menor que N si alguna línea está malformada o si EOF llega antes de leer todas — el motor no aborta el lote, sólo salta esa línea.

Si la cabecera está malformada (intensidad o N no parseables), el comando se ignora silenciosamente y NO se consumen líneas adicionales — comportamiento consistente con los demás comandos no reconocidos del protocolo.

Decisiones tomadas durante la implementación

  1. Coordenadas enteras, no float. IGNITE acepta float y trunca; aquí recibimos enteros directamente. La brocha de Godot trabaja en celdas, y la API enviará pares [x,z] enteros. Mantener floats sería ruido.

  2. Validar wx<0 || wz<0 explícitamente. La división entera de C++ trunca hacia cero (no hacia menos infinito), así que wx=-1 → qx=0 pasaría el check de cuadrante pero accedería a idx=-1 en fuel_coefficient. Bug heredado de ignite(); aquí lo hago bien desde el principio. Considerar parchearlo en ignite() cuando toque.

  3. Clamp de intensidad a [0,1] en el motor. Paranoia barata: si alguien manda intensidad=2 por stdin, no convertimos fuel en negativo (que arrancaría state == AIR con fuel_coefficient < 0).

  4. Getter público fuel_modificado() const. Necesario para tests y para que la issue #70 pueda iterarlo en /reset. No expone mutación.
