Hallazgo: el `state` SMOKE no se propaga verticalmente en el motor

← Inicio · Pipeline de Simulación · Integracion Godot-Motor — Recopilatorio

Milestone afectado: Integración Godot-Motor (#79, #92) Archivos relacionados: engine/adveccion.h · engine/reaccion.h · engine/simulacion.h

No es estrictamente un bug, sino una limitación arquitectónica del solver que descubrimos al intentar visualizar la columna de humo. Documentado porque marcó dos decisiones del milestone (#92 y #94) y la respuesta directa de Claude a la sospecha del usuario fue errónea inicialmente — propaga aprendizaje.

El síntoma

Al implementar _aplicar_diffs() en Godot tras el refactor a MultiMesh (#79), el usuario reportó:

"Sigue sin existir columna de humo. El fuego ahora se pinta un poco por encima del suelo. ¿Podemos añadir debug a ver porque no existe columna como tal? A ver si el motor no produce o esque godot lo siega o alomejor es la API traduciendo incorrectamente."

Yo (Claude) afirmé inicialmente que el motor sí calculaba la columna y que Godot la estaba ignorando aplastando la coordenada Y. Estaba parcialmente equivocado.

El usuario añadió debug en _aplicar_diffs y descubrió que:

[DBG 55] diffs(475): BURNING=252 SMOKE=223
  y_BURN[min=10 max=12 avg=11.0]
  y_SMOKE[min=10 max=12 avg=11.0]

y_SMOKE y y_BURN están en el mismo rango: ambas se quedan en la fila y_superficie del subgrid (~nivel 11). No hay estado SMOKE por encima del suelo.

Por qué pasa (lo que el solver hace)

Tras releer engine/adveccion.h y engine/reaccion.h:

  1. aplicarBuoyancy() (engine/buoyancy.h): cpp if (c.state != BURNING && c.state != SMOKE) continue; float fuerza = A * c.temperature - B * c.density; c.velocity.z += fuerza; Aplica empuje vertical sólo a celdas BURNING o SMOKE. Para celdas AIR (la mayoría) no hace nada.

  2. advectar() (engine/adveccion.h) semi-Lagrangiano MacCormack: cpp dst.temperature = semiLagrangiano(g, g, x, y, z, dt, getTemp); dst.density = semiLagrangiano(g, g, x, y, z, dt, getDensity); dst.reaction = semiLagrangiano(g, g, x, y, z, dt, getReaction); Transporta temperature, density, reaction entre celdas adyacentes. NO transporta state.

  3. aplicarReaccion() (engine/reaccion.h) — el único lugar donde state cambia: cpp if (c.state == FUEL && c.temperature > UMBRAL_IGNICION) { c.state = BURNING; ... } if (c.state == BURNING && c.reaction <= 0) { c.state = SMOKE; } Sólo dos transiciones: FUEL→BURNING y BURNING→SMOKE. No existe AIR→SMOKE por mucho calor que reciba una celda AIR.

  4. Las celdas FUEL sólo se inicializan en y_superficie (la fila del DEM): cpp // engine/simulacion.h::activar for (int cz = 0; cz < CELDAS_QUAD_Z; ++cz) { for (int cx = 0; cx < CELDAS_QUAD_X; ++cx) { const int v_sup = c.y_superficie[idx]; Celda& sup = c.grid.get(cx, cz, v_sup); sup.state = EstadoCelda::FUEL; } } Por tanto BURNING (que sólo viene de FUEL) y SMOKE (que sólo viene de BURNING) únicamente pueden existir en y_superficie.

Resumen físico

  • El motor SÍ calcula que el aire encima del foco se calienta (advección + buoyancy transportan temperatura).
  • El motor NO marca ese aire como SMOKE — sigue siendo AIR con T alta.
  • La API sólo serializa estados ∈ {BURNING, SMOKE} (api/state.py::_RENDERABLES).
  • La columna física existe en los números, no en el state.

El error de Claude al diagnosticar

Mi primera respuesta fue:

"El motor SÍ propaga humo verticalmente — la advección lleva temperatura/density por velocity, así que el humo sube por buoyancy y se transporta."

Esto mezclaba dos cosas distintas:

  • Verdad: temperature y density sí se transportan vertical.
  • Falsedad implícita: que el state SMOKE acompañaba a esa transferencia.

El usuario me corrigió empíricamente con el print del cliente, que mostraba y_SMOKE y y_BURN en el mismo rango. Tuve que leer 30 líneas más de adveccion.h para darme cuenta.

Disculpa documentada en el commit 38c0f58:

"En el commit 003c820 te vendí 'humo en altura real del motor'. Lo escribí confiado en que el motor propagaba state verticalmente. Hubiera bastado con leer 30 líneas más de adveccion.h para darme cuenta."

Las dos decisiones del milestone que salen de aquí

#92 — SMOKE virtual

Sin tocar el solver, en el bucle de emisión de engine/main.cpp añadir una función estado_aparente():

inline EstadoCelda estado_aparente(const Celda& c) {
    if (c.state == EstadoCelda::AIR &&
        c.temperature > UMBRAL_SMOKE_VIRTUAL_TEMP) {  // 200°C
        return EstadoCelda::SMOKE;
    }
    return c.state;
}

Reporta como SMOKE las celdas AIR calientes. El state real nunca se modifica, sólo el reportado. El solver es idéntico, sólo cambia lo que llega al cliente.

Coste documentado: ×2.4 latencia, ×58 diffs (todo I/O motor→Python, las fases del solver no se ralentizan).

#94 — Toggle

El coste de la columna virtual sólo se paga cuando el usuario quiere verla. Comando SMOKE_VIRTUAL on|off en el motor, endpoint POST /humo_virtual en la API, tecla H en el cliente. Con humo OFF el bench vuelve a baseline (~250 ms/tick).

Limitaciones inherentes del modelo

Esta arquitectura del motor (state discreto + 20 niveles verticales × ~7.68 m) tiene límites físicos asumidos:

  • La columna del motor llega máximo a 146 m sobre el DEM.
  • En un incendio real una columna pirocumulus puede alcanzar varios km de altura.
  • Los niveles bajos del subgrid donde el suelo es alto (>1700 m) están dentro del DEM y nunca contendrán humo.

Para un TFG con foco en propagación lateral y planificación de cortafuegos, estos límites son aceptables. Para realismo atmosférico haría falta un grid plenamente 3D (no 20 niveles locales por cuadrante) y un solver con state continuo en lugar de discreto. Fuera del scope.

Lección general

Antes de afirmar "el motor calcula X", leer las líneas reales del solver. El nombre de la variable (state, temperature, density) no implica que esa variable participe en todos los pasos del pipeline.

Y la lección operacional:

El usuario tenía la sospecha correcta ("¿el motor no calcula la flotabilidad o sí?"). Yo (Claude) confirmé prematuramente. La validación empírica del usuario (un print en el cliente) fue más rápida que mi lectura del código.

Cuando el usuario sospecha algo concreto sobre el motor, leer el código del motor antes de afirmar nada.
