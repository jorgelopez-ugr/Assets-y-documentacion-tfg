`evaluar_desactivacion` pisaba los halos del vecino

← Inicio · simulacion · Pipeline de Simulación

Síntoma

En el test E2E del rediseño (api/test_e2e.py), encender una celda en el borde derecho del cuadrante (5, 5) no propagaba el fuego al cuadrante (6, 5) aunque corriéramos 20 ticks. El test fallaba con:

AssertionError: el fuego NO cruzó al cuadrante (6,5) en 20 ticks.
distribución por cuadrante: 5_5=2394, 6_5=0, total=2394

Curiosamente, sí se observaba que 9 cuadrantes (todos los vecinos Moore del 5,5 más él mismo) acababan activos según TICK_DONE 450000, pero ninguno de ellos tenía celdas BURNING — solo FUEL.

Análisis

El flujo de un tick original era:

1. física por cuadrante (incluye sostener_burning)
2. propagar_halos
3. evaluar_activacion       ← activa vecinos con calor en frontera
4. evaluar_desactivacion    ← libera cuadrantes sin BURNING

Tick a tick:

  • Tick 1: A activo. evaluar_activacion ve temp = 800 °C en el borde de A → activa a B (y a los demás vecinos por simetría). B nace con FUEL/AIR pero sin BURNING todavía.

  • Tick 2: Física en A y B. propagar_halos copia A's right column ↔ B's left column, promediando temperaturas: A's (49,25,sup) con T=800 y B's (0,25,sup) con T=0 quedan ambas a T=400. Bien.

Después: evaluar_desactivacion recorre activos. B no tiene BURNING → desactiva B. Al desactivar, c.grid.cells.clear() libera el SubGrid. Los halos recién recibidos se pierden.

Después: evaluar_activacion vuelve a ver el borde de A caliente (también promediado a 400, todavía > 100 → reactiva B. Pero activar llama a inicializar() que hace cells.assign(total, Celda{}) — reset completo. B vuelve a nacer con FUEL/AIR fríos.

El ciclo activar → desactivar → activar pisa los halos. El calor nunca llega a los FUEL de B y nunca ignita.

Arreglo

Dos cambios en engine/simulacion.h:

1. Cambiar el orden: desactivar antes de activar

// Antes:
evaluar_activacion();
evaluar_desactivacion();

// Después:
evaluar_desactivacion();
evaluar_activacion();

Así, un cuadrante recién activado en este tick no es candidato a desactivación hasta el siguiente. Se le da un tick de gracia para recibir halos sin perderlos.

2. Considerar "actividad" la temperatura, no solo BURNING

void evaluar_desactivacion() {
    for (uint32_t id : activos_) {
        bool actividad = false;
        for (const Celda& cell : c.grid.cells) {
            if (cell.state == EstadoCelda::BURNING ||
                cell.temperature > UMBRAL_ACTIVACION) {  // ← clave
                actividad = true; break;
            }
        }
        if (!actividad) /* desactivar */;
    }
}

Esto es la pieza esencial. Un cuadrante recién activado con halos calientes (temp > 100 °C) es "activo" aunque no tenga BURNING todavía. Cuando la reacción del siguiente tick lo ignite, se convertirá en BURNING. Si no lo ignita (calor insuficiente), pasará a desactivarse en algún tick futuro naturalmente.

Cómo lo cazó el E2E

El test E2E reveló el bug porque era el primer test que probaba la propagación entre cuadrantes end-to-end. Los tests unitarios de simulacion.h validaban activación, desactivación, halos por separado y todos pasaban — pero su composición tenía esta interacción patológica.

Lección: los tests unitarios no detectan errores de orden global. Hace falta al menos un E2E que ejecute el sistema completo durante varios ticks para validar la coherencia temporal.

Lecciones generales

  1. El orden de las fases de un tick no es libre. Si un paso modifica estado que otro consume, el orden tiene que reflejar la dependencia.
  2. Liberar memoria es destructivo. cells.clear() no es reversible: los halos copiados allí se pierden. Toda decisión de liberar debe verificar que no quede información útil.
  3. "Actividad" es más que BURNING. En un sistema con halos, una celda caliente sin combustible aún propaga calor a sus vecinos. La métrica de "está vivo" tiene que cubrir ambos casos.
