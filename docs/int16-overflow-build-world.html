Error: int16 overflow al calcular distancia² RGB en cargar_fuel_map

← Inicio · build-world-leyendo-especies · Integracion Godot-Motor — Recopilatorio

Milestone afectado: Integración Godot-Motor (Fase 2 #81/#82) Archivos involucrados: tools/build_world.py

El síntoma

Al implementar cargar_fuel_map() en tools/build_world.py para leer las 7 capas hiperespectrales de docs/references/color-*.png, los porcentajes de píxeles matcheados salían absurdos:

[FUEL] color-terreno.png         matched 2098953 celdas (94.3%)
[FUEL] color-vegetacion-alta.png matched 2147378 celdas (96.5%)
[FUEL] color-vegetacion-baja.png matched 2170267 celdas (97.5%)
[FUEL] color-pinus-helepensis.png matched 2073856 celdas (93.2%)
[FUEL] color-encina.png          matched 2224838 celdas (99.9%)
[FUEL] color-quejigo.png         matched 2223144 celdas (99.9%)
[FUEL] color-camino.png          matched 2072566 celdas (93.1%)

Cada categoría matcheaba casi todo el mapa independientemente del color objetivo. Los colores agregados (verde, azul, rojo) podrían pasar por colisión con la ortofoto natural, pero el cian de Pinus matcheando 93 % del DEM o el violeta de encina matcheando 99.9 % son físicamente imposibles.

Comprobación independiente del mismo cálculo, sobre la imagen original sin resize:

color-terreno.png   tol=10  matched 125337 (21.98%)
color-encina.png    tol=10  matched 40331  (7.07%)

→ Esos números sí son razonables. ¿Por qué tras resize el ratio se dispara?

Por qué pasaba

El código que matcheaba colores era:

pixels = np.asarray(img, dtype=np.int16)
diff = pixels - np.array(color_esperado, dtype=np.int16)
dist2 = (diff[..., 0] ** 2 + diff[..., 1] ** 2 + diff[..., 2] ** 2)
mascara = dist2 <= (COLOR_TOLERANCIA ** 2)

pixels está en int16. int16 tiene rango [-32768, 32767].

Para componentes RGB en [0, 255], la diferencia diff está en [-255, 255]. Hasta aquí, bien.

Pero 255 ** 2 = 65025, que sobrepasa el rango de int16. NumPy preserva el dtype y hace el cuadrado con overflow silencioso. Resultado:

>>> np.int16(255) ** 2
np.int16(-535)
>>> np.int16(200) ** 2
np.int16(-25536)

Cuando sumamos tres componentes cuadradas con overflow, dist2 sale en [-32768, 32767] con valores negativos para casi todos los píxeles cuyos componentes diff son grandes.

Y dist2 <= 100 con dist2 negativo es trivialmente True. La máscara cubría ~99 % del mapa.

Por qué el debug independiente sí funcionaba

El script suelto que validé antes del bug iteraba uno a uno por componente y comparaba: usaba int(r) y int(g) y int(b) de Python (precisión arbitraria), no array NumPy. Por eso reportaba ratios correctos.

Tras meterlo en el flujo de cargar_fuel_map con NumPy y un dtype explícito, el overflow apareció. Lección: el dtype del array NumPy es contagioso a las operaciones. arr16 ** 2 sigue siendo int16.

La solución

Una sola línea:

pixels = np.asarray(img, dtype=np.int32)   # antes: int16
diff = pixels - np.array(color_esperado, dtype=np.int32)

int32 cubre [-2_147_483_648, 2_147_483_647]. Tres componentes cuadradas suman como mucho 3 × 65025 = 195_075. Encaja sin problema.

Distribución física coherente con la imagen visible.

Test que cazaría esto

engine/test_world_loader.cpp::test_fuel_valores_canon falla si algún byte del world.dat cae fuera del set {0.0, 0.4, 0.5, 0.55, 0.9, 0.95}. Con el overflow, casi todas las celdas terminaban con fuel=0.5 (último archivo no-cero procesado, que es quejigo) — el test capturaría el patrón uniforme con test_fuel_no_uniforme (que exige ≥2 valores distintos).

Lección general

Cuando un cálculo numérico te dé un resultado que va contra la intuición física, comprueba el dtype antes que la lógica. NumPy hereda dtype en operaciones aritméticas; un int16 multiplicado consigo mismo sigue siendo int16. Si el resultado puede exceder el rango, hay que upcastear explícitamente.

Para tareas de procesado de imágenes con NumPy: siempre arrancar en int32 o float32. int8/int16 ahorran memoria pero son trampolines de overflow.
