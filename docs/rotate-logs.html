rotate_logs.py — rotación manual de logs

← Inicio · Sistema de Logging

⛔ OBSOLETO (2026-06-15)

El subsistema de logging quedó en desuso. rotate_logs.py y su test se sacaron del control de versiones el 2026-06-15 (la carpeta logs/ entera está ahora en .gitignore). El script sigue en disco localmente, pero ya no forma parte del repositorio. Nota conservada como referencia histórica.

Propósito

Los ficheros .log crecen sin límite si nadie los administra. Este script comprime los logs que superan los umbrales configurables y los guarda con fecha para que sea fácil encontrarlos. El fichero original se vacía pero no se elimina, para que los handlers de logging activos no rompan al intentar escribir en él.

Uso

# Desde la raíz del proyecto
python logs/rotate_logs.py

# Ver qué se rotaría sin modificar nada
python logs/rotate_logs.py --dry-run

Salida típica:

Rotación de logs en: /ruta/al/proyecto/logs
  umbral tamaño : 1 MB
  umbral edad   : 7 días

  sin cambios : engine.log
  rotando     : api.log → api.log.2026-04-17.gz  (1.23 MB ≥ 1 MB)

1 fichero(s) rotado(s).

Umbrales configurables

Al principio del script hay dos constantes:

MAX_SIZE_MB  = 1    # comprimir si el fichero supera este tamaño en MB
MAX_AGE_DAYS = 7    # comprimir si el fichero tiene más de estos días

Un fichero se rota si cumple cualquiera de los dos criterios. Los ficheros vacíos nunca se rotan.

Nombre del fichero comprimido

api.log  →  api.log.2026-04-17.gz

Si ya existe un .gz con la misma fecha (por ejemplo, si el script se ejecuta dos veces el mismo día), se añade un sufijo numérico:

api.log.2026-04-17.gz      ← ya existía
api.log.2026-04-17.1.gz    ← nuevo
api.log.2026-04-17.2.gz    ← si vuelve a ejecutarse

Por qué truncar en lugar de eliminar

Si se eliminara el .log activo, el RotatingFileHandler de la API perdería el descriptor de fichero y dejaría de escribir logs hasta el siguiente reinicio. Truncar (vaciar a 0 bytes) mantiene el descriptor válido — los nuevos mensajes se escriben desde el principio del fichero vacío.

Esta misma razón hace que no se deba usar este script mientras la API está en producción sin precaución. Lo correcto es ejecutarlo durante una ventana de mantenimiento o configurar un cronjob de madrugada.

Cronjob (futuro)

El script incluye un ejemplo comentado al principio:

# 0 3 * * * /usr/bin/python3 /ruta/proyecto/logs/rotate_logs.py

Esto ejecutaría la rotación todos los días a las 3:00. Por el momento la rotación es manual.

Relación con el RotatingFileHandler

La API tiene su propio mecanismo de rotación interno (5 MB automáticos, 3 backups). rotate_logs.py es complementario:

RotatingFileHandler rotate_logs.py
Cuándo actúa Durante la sesión, automático Manual / cronjob
Qué hace Renombra a .log.1, .log.2... Comprime con fecha .gz
Backups 3 ficheros sin fecha Indefinidos con fecha
Requiere reinicio No No (truncate)

Para un despliegue real, los dos mecanismos conviven: el handler gestiona el crecimiento intradiario y el script archiva los .log.1 / .log.2 con fecha.

Tests

cd /ruta/proyecto
api/.venv/bin/pytest logs/test_rotate_logs.py -v   # 6 tests

Casos cubiertos: rotación por tamaño, por antigüedad, sin rotación, log vacío, colisión de fecha con sufijo numérico, dry-run.
