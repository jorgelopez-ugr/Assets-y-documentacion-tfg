Arquitectura de la API

← Inicio · Arquitectura del Sistema · Protocolo del Motor

Responsabilidad

La API es un puente y orquestador, sin lógica de física. Sus responsabilidades:

  1. Mantener vivo el proceso C++ del motor (lazy spawn) y traducir HTTP ↔ comandos de texto.
  2. Servir los diffs de cada tick a Godot (protocolo basado en cambios).
  3. Orquestar el flujo del producto: cortafuegos (/firebreak), trazado A* (/trazar) y evaluación (/evaluar, /puntuar, /resultado).

El pathfinding (A*) y la evaluación viven en Python (no son física); el motor C++ no sabe de red ni de caminos. Ver Trazado automático A — coste y datos (M3) y Evaluación de caminos (M4).

Módulos

  • motor.py — wrapper del proceso C++ (lazy Popen, un _lock serializa el acceso). Expone: ignite, tick→diffs, reset, set_physics_mode/ get_physics_mode, pintar_firebreak/deshacer_firebreak, set_viento, set_humo_virtual, quadrants, stats, fuel_map, shutdown.
  • state.py — acumulador en memoria de celdas renderables. update_from_changed aplica los CHANGED y devuelve los diffs renderables (transiciones BURNING/SMOKE; los cambios de solo-temperatura se filtran). active_cells() para el /snapshot. clear().
  • dem.py — elevación/pendiente fina por celda (lee el heightmap).
  • pathfinding.py — A* sobre el DEM + coste por celda (CosteParams).
  • evaluacion.py — zona protegida + métrica compuesta puntuar().
  • resultado.py — sesión de evaluación en vivo (acumula factores por tick).
  • main.py — los endpoints HTTP. lifespan no carga estado (world.dat lo gestiona el motor); solo cierra el proceso al apagar.

Endpoints

Simulación

Endpoint Descripción
POST /reset {physics_mode?} Limpia el estado dinámico; opcional conmutar surface/volumetric. Devuelve el modo activo.
POST /ignite {x,z} Enciende la celda de superficie en (x,z) metros.
POST /tick Avanza un tick. Devuelve {"changed": [diffs]} (solo celdas que cambiaron a/desde BURNING/SMOKE).
GET /snapshot Foto completa de celdas BURNING/SMOKE (sync inicial de un cliente nuevo).
GET /status Modo de física activo ({"physics_mode": ...}).

Protocolo de diffs (MVP7): /tick devuelve los cambios del último tick, no el estado completo. /snapshot solo se usa para sincronizar al conectar. (Antes existía un GET /active_cells que se sondeaba cada frame; quedó obsoleto.)

Cortafuegos y trazado

Endpoint Descripción
POST /firebreak {cells, intensidad} Pinta cortafuegos (anula fuel, terraforma, estado CORTAFUEGOS). → {painted}
DELETE /firebreak {cells} Deshace cortafuegos. → {undone}
POST /trazar {origen, destino, pesos?} A* entre 2 puntos → vector de celdas (el cliente lo envía a /firebreak).
GET /trazar/defaults Pesos default del A* (CosteParams); el HUD los pide al arrancar para inicializar sus sliders (M8).

Evaluación (el fuego como métrica)

Endpoint Descripción
POST /evaluar {focos, firebreak_cells?, ticks} Simula y recolecta factores (área, contención, tiempo).
POST /puntuar {focos, firebreak_cells, ticks, pesos?} Métrica compuesta ponderada (batch) → puntuación + desglose.
POST /resultado {pesos?} Puntúa el incendio REAL de la sesión (factores acumulados en vivo).

Otros

POST /viento {vx,vz} · POST /humo_virtual {activo} · GET /fuel_map · GET /quadrants · GET /stats · GET / (health).

Flujo típico (3 fases) desde Godot

[arranque]  (motor aún no lanzado)
1. PINTAR   POST /firebreak  (manual)  o  POST /trazar → POST /firebreak (auto)
2. QUEMA    POST /ignite     ← motor se lanza aquí, carga world.dat
            bucle:  POST /tick  → {changed:[...]}  (Godot pinta los diffs)
3. EVALUAR  (5 s sin fuego o 5 min)  POST /resultado → puntuación + desglose
[reset]     POST /reset

conftest.py compila el motor si hace falta y aísla el estado entre tests (motor.shutdown() + state.clear() + resultado.reset_sesion()).
