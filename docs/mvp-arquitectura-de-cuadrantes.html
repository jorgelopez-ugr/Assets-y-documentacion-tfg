MVP — Arquitectura de Cuadrantes

⚠ Documento histórico (pre-refactor)

Refleja un estado anterior al refactor de caminos forestales (jun 2026); se conserva como traza del proceso. El estado actual está en Inicio.

← Inicio

Objetivo

Rediseño arquitectónico completo del motor. El sistema anterior tenía un grid monolítico de 100×10×100 (~16 m² por celda), procesaba todas las celdas en cada tick y enviaba el estado completo en JSON. El rediseño:

  • Trocea el terreno en cuadrantes activables dinámicamente.
  • Sube la resolución a 1×1 m horizontal (50.000 celdas por cuadrante).
  • Sustituye el JSON completo por un protocolo de comandos texto con CHANGED solo para celdas que han cambiado.
  • Precomputa offline todo lo estático del terreno en data/world.dat.

No añade features visibles para el usuario final — corrige el modelo mental y desbloquea simulaciones grandes sin hacer estallar la red ni la RAM.

La arquitectura nueva

[OFFLINE — una vez]
heightmap_16bit.png
        ↓
[[build_world]]  →  data/world.dat   (binario, ~11 MB, versionado)

[RUNTIME]
API arranca  →  motor C++ se lanza al primer comando, carga world.dat
        ↓
POST /ignite       →  IGNITE wx wz       →  activa cuadrante, BURNING
POST /tick         →  TICK                →  procesa solo activos
                                          →  CHANGED + TICK_DONE
GET /active_cells  →  acumulador en memoria filtra a BURNING + SMOKE
POST /reset        →  RESET               →  limpia estado dinámico

Detalle: Arquitectura del Sistema · Cuadrante y Celda · Pipeline de Simulación · Protocolo del Motor.

#40 — engine/simulacion.h

El corazón del nuevo motor. Clase Simulacion con activar, ignite, tick, reset, y los métodos privados propagar_halos, evaluar_activacion, evaluar_desactivacion, sostener_burning.

Decisiones críticas:

  • Orden de un tick: física por cuadrante → halos → desactivar → activar. Desactivar antes de activar para que vecinos recién encendidos sobrevivan al tick.
  • Una celda BURNING es un emisor sostenido: tras la advección que la drena, sostener_burning reinyecta TEMP_IGNICION = 800 °C.
  • Un cuadrante con temp > UMBRAL_ACTIVACION (100 °C) cuenta como "activo" aunque no tenga BURNING — necesario para que un vecino recién recibido por halo no se desactive antes de ignitar.

Tests: 18/18 — engine/test_simulacion.{cpp,sh}.

Ver: simulacion · Pipeline de Simulación · evaluar-desactivacion-pisaba-halos.

#41 — engine/main.cpp — bucle stdin

El binario del motor. Carga data/world.dat, construye Simulacion, y entra en bucle leyendo comandos por stdin.

while (getline(cin, linea)) {
    if (cmd == "IGNITE") sim.ignite(wx, wz);
    if (cmd == "TICK")   {
        sim.tick();
        // emitir CHANGED para celdas con (state, temp) distintos
        // del último valor emitido
        cout << "TICK_DONE " << total << "\n" << flush;
    }
    if (cmd == "RESET")  sim.reset();
    if (cmd == "QUIT")   break;
}

Caché por celda: unordered_map<uint64_t, LastEmit> con UMBRAL_TEMP_CHANGED = 0.5 °C. Solo viajan las celdas cuyo estado o temperatura han cambiado.

#42 — api/motor.py + state.py + main.py

Adaptación de la API al nuevo protocolo.

Reescritos por completo:

  • motor.py: wrapper de subprocess.Popen con ignite, tick, reset, shutdown. Parser de CHANGED ... TICK_DONE.

Nuevo:

  • conftest.py: compila el motor antes de la sesión de tests si está desactualizado, y aísla el estado entre tests.

#43 — Verificación end-to-end

api/test_e2e.py arranca la API con TestClient, ignita en el borde de un cuadrante (coordenada (299.5, 275.5) = celda (49, 25) del cuadrante (5, 5)), corre 20 ticks y verifica que aparecen celdas con x ≥ 300 — prueba que el fuego ha cruzado al cuadrante (6, 5).

Bug encontrado y arreglado en este E2E (en simulacion.h): evaluar_desactivacion desactivaba el vecino recién activado por halo, y al re-activarse se reinicializaba el SubGrid perdiendo el calor del halo. Ver evaluar-desactivacion-pisaba-halos.

Lo que NO entró en este MVP

  • Refactor de adveccion.h/presion.h: siguen tomando Grid& (typedef de SubGrid) y haciendo 3 copias de SubGrid por tick por cuadrante (~5 MB cada una). Refactorizable a operar in-place o sobre scratch buffers reutilizables.
  • Refactor a hablar de Cuadrante directamente: las cabeceras de física aún reciben Grid&. Es deuda visible pero funcional.
  • Estado HALO real: la enum EstadoCuadrante::HALO existe pero no se usa. La idea futura: cuadrantes que solo procesen su columna de borde cuando un vecino activo esté ardiendo.
  • Integración hiperespectral: fuel_coefficient sigue hardcoded a 1.0 para todo el DEM.
  • Caminos forestales / cortafuegos: requerirían vectores con fuel_coefficient = 0.0 específicos.
  • Optimización GPU: el motor sigue mono-hilo en CPU. A 1–10 cuadrantes activos no se nota; con incendios mayores podría hacer falta.
  • Godot: integrar /active_cells con la nueva resolución 1×1 m queda como responsabilidad del usuario.
