Por qué esta issue existe

El milestone se mide en latencias antes/después por PR. Antes de tocar nada del solver hace falta un instrumento que dé números comparables: si el ruido entre dos ejecuciones consecutivas es del 30%, ningún PR puede demostrar mejora menor del 30%. Y necesitamos un formato que se pueda adjuntar al PR sin tener que copiar logs a mano.

Qué se ha cambiado

api/headless_bench.py

  1. Flag nuevo --csv path.csv que vuelca una fila por tick con:

tick,latency_ms,n_diffs,burning,smoke

burning y smoke se rellenan sólo en los ticks de snapshot (cada 20 por defecto). En el resto van vacíos para no inflar el coste del bench pidiendo /snapshot cada tick.

  1. Docstring del módulo documenta: - Determinismo: el motor no usa RNG; el terreno (world.dat) fija el fuel y la ignición fija el origen. Por eso no hay flag --seed. - Comando estándar de comparación que hay que ejecutar en cada PR del milestone:

    bash python3 api/headless_bench.py --ticks 200 --csv bench_<rama>.csv

  2. La lógica de escritura CSV vive en una función pura escribir_csv(filas, destino) para que sea testeable sin tener la API arrancada. Acepta path o file-like (StringIO).

api/test_headless_bench.py (nuevo)

Cuatro tests sobre escribir_csv: - cabecera y orden de columnas fijos, - fila completa serializada correctamente, - None en burning/smoke se convierte en cadena vacía, - acepta path como destino.

El bucle principal del bench requiere la API arrancada y queda fuera de pytest.

Verificación del criterio de aceptación

Criterio: dos ejecuciones consecutivas con la misma ignición producen latencias cuya diferencia media < 5%.

Ejecutado dos veces con --ticks 80 --csv /tmp/run{1,2}.csv (80 ticks cubren ya parte de la fase de crecimiento exponencial — pico de 1410 BURNING):

Métrica Run 1 Run 2
Tiempo total 10.08 s 9.97 s
Latencia media /tick 124.19 ms 122.89 ms
n_diffs por tick idéntico en los 80 ticks idéntico
BURNING/SMOKE en cada snapshot idénticos idénticos

Diferencia relativa de medias: 1.06% — bajo el umbral del 5%.

La simulación es 100% determinista a nivel de estado (verificado: todas las filas de n_diffs, burning y smoke coinciden bit a bit entre runs). El único ruido es el de scheduling/cachés del SO sobre los tiempos de pared, y queda muy por debajo del 5%.

Baseline registrado

Hardware: Ryzen 7 5800H, motor monohilo, sin Godot conectado.

Para 80 ticks, ignición central: - Latencia media /tick: ~123 ms - p50: ~70 ms · p95: ~400 ms · max: ~430 ms - Pico en tick 80: 1410 BURNING / 10183 SMOKE

Para los hitos del milestone (de bench previo del 9-may-2026): - Tick 100 (~2651 BURNING): 726 ms - Tick 140 (~25593 BURNING): 4.1 s - Tick 160 (~52390 BURNING): 10.0 s

Estos son los números a batir.
