TL;DR

  • Tercera pieza del milestone "Cortafuegos pintables desde Godot": expone por HTTP los dos comandos del motor introducidos en #69 y #70.
  • POST /firebreak: pinta una lista de celdas con una intensidad ∈ [0, 1]. Default 1.0. Respuesta {status, painted: N}.
  • DELETE /firebreak: deshace cortafuegos previos en una lista de celdas. Respuesta {status, undone: N}.
  • Validación con pydantic: intensidad fuera de rango → 422; cells malformado (no lista, pares de longitud incorrecta, no enteros) → 422.
  • No toca el motor: sólo capa HTTP. La API pública existente (/tick, /snapshot, /ignite, /reset) no cambia su contrato.
  • Cero regresiones: 31 motor + 74 motor + 61 API (era 46, +15).

Por qué esta issue es la siguiente

El motor ya tenía las dos piezas de la mecánica (#69 + #70). Sin la API no hay forma de invocarlas desde fuera del proceso C++ — Godot habla con HTTP, no con stdin/stdout del motor. Esta issue es el puente.

Una vez cerrada, Godot puede: 1. Pintar con la brocha del ratón. 2. Enviar POST /firebreak {cells, intensidad} por trazo. 3. Enviar DELETE /firebreak {cells} para borrar (futuro: undo del cliente o "borrar franja" como herramienta de pincel inverso).

Falta sólo la validación narrativa end-to-end (#72).

api/motor.py — wrappers nuevos

def pintar_firebreak(cells: list[tuple[int, int]],
                    intensidad: float = 1.0) -> int:
    p = _proceso_activo()
    n = len(cells)
    p.stdin.write(f"FIREBREAK_BATCH {intensidad} {n}\n")
    for x, z in cells:
        p.stdin.write(f"{x} {z}\n")
    p.stdin.flush()
    while True:
        linea = p.stdout.readline().strip()
        if linea.startswith("FIREBREAK_DONE"):
            return int(linea.split()[1])

api/main.py — modelos y endpoints

class FirebreakPayload(BaseModel):
    cells: list[tuple[int, int]]
    intensidad: float = Field(default=1.0, ge=0.0, le=1.0)

class FirebreakUndoPayload(BaseModel):
    cells: list[tuple[int, int]]


@app.post("/firebreak")
def firebreak(payload: FirebreakPayload):
    n = motor.pintar_firebreak(payload.cells, payload.intensidad)
    return {"status": "ok", "painted": n}


@app.delete("/firebreak")
def firebreak_undo(payload: FirebreakUndoPayload = Body(...)):
    n = motor.deshacer_firebreak(payload.cells)
    return {"status": "ok", "undone": n}

Body(...) explícito en DELETE: FastAPI requiere indicar que el cuerpo JSON es obligatorio porque DELETE con body no es totalmente estándar (aunque está permitido por la RFC).

Decisiones tomadas durante la implementación

  1. tuple[int, int] en lugar de list[int]. Pydantic v2 lo interpreta como "array de exactamente 2 enteros" en JSON, así que [1, 2] pasa pero [1, 2, 3] se rechaza con 422 sin validación custom. La alternativa habría sido conlist(int, min_length=2, max_length=2) — más verboso y equivalente.

  2. DELETE con cuerpo, no query string. La alternativa sería DELETE /firebreak?cells=... pero pasar listas de cientos de pares por query string es un dolor (URL-encoded, longitud máxima, parseo). El cuerpo JSON es lo que ya envía el cliente Godot para POST y es lo más natural para reusar el pipeline.

  3. El wrapper devuelve int, no bool. El motor reporta FIREBREAK_DONE N con N celdas procesadas — útil para verificar en tests que el motor recibió todo el lote. La API expone ese N en el campo painted/undone de la respuesta, no como simple "ok".

  4. Validación funcional ligera en este test, no exhaustiva. Pintar la celda vecina al foco y comprobar que tras 1 tick no es BURNING demuestra que el contrato funciona end-to-end. La validación narrativa completa (franja estrecha vs ancha, demostración del TFG) queda para #72.

  5. /reset borra cortafuegos sin DELETE explícito. Es un efecto secundario directo de #70 (reset() restaura miguitas), pero merece un test en la API porque es comportamiento observable por Godot. El test test_firebreak_reset_borra_efecto lo verifica.
