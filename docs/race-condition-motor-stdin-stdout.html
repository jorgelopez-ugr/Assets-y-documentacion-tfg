Error: race condition en el stdin/stdout del motor desde threads de FastAPI

← Inicio · Integracion Godot-Motor — Recopilatorio · Protocolo del Motor

Milestone afectado: Integración Godot-Motor Issue / PR: se descubre validando #88+#89 (overlay Q), se arregla en PR #101 Archivos involucrados: api/motor.py

El síntoma

Durante la validación del overlay de cuadrantes activos (tecla Q), los logs de uvicorn empezaron a llenarse de excepciones del parser de tick():

ValueError: could not convert string to float: '6.211.3827'
ValueError: could not convert string to float: '13.2-22.6391'
ValueError: could not convert string to float: '-13.3378.0248'
INFO: 127.0.0.1:37814 - "POST /tick HTTP/1.1" 500 Internal Server Error

Cada cadena '6.211.3827' es claramente dos floats pegados: 6.21 y 1.3827. El parser de motor.py::tick() esperaba 6 campos en la línea CHANGED y, al fallar, devolvía 500 al cliente.

En el lado de Godot, los efectos eran intermitentes y confusos:

  • La tecla Q parecía togglear sola pasado un rato.
  • El toggle de humo (H) dejaba de responder.
  • Las celdas BURNING se acumulaban sin actualizarse.

Por qué pasaba

El motor C++ es un único proceso lanzado con Popen que se comunica con la API por stdin/stdout (decisión arquitectónica fundamental, ver Protocolo del Motor).

FastAPI corre cada endpoint en un thread del threadpool de anyio. Los endpoints que tocan el motor (/tick, /quadrants, /ignite, /humo_virtual, …) compiten por el mismo stdin/stdout:

Thread A (POST /tick):
  motor.stdin.write("TICK\n")
  motor.stdin.flush()
  while True:
      linea = motor.stdout.readline()   ← está leyendo CHANGEDs

Thread B (GET /quadrants disparado por el Timer del overlay Q):
  motor.stdin.write("QUADRANTS\n")     ← INYECTA en medio
  motor.stdin.flush()
  linea = motor.stdout.readline()

El motor recibe TICK\n + QUADRANTS\n casi a la vez. Procesa TICK, emite muchísimas líneas CHANGED. En medio del flujo, ATIENDE la línea QUADRANTS y emite una línea QUADRANTS qx=32 .... Los dos readline() (de Thread A y Thread B) compiten por las líneas del buffer y se llevan lo que pueden.

Resultado catastrófico: si el \n final de una CHANGED se pierde en el buffering, el readline() lee dos CHANGEDs pegados en una sola línea. El parser ve 11 tokens en lugar de 6, los tokens 5 (state) y 6 (temperature) salen con dos floats concatenados.

La trampa narrativa

Lo más confuso es que el bug se manifestaba como problemas visuales, no como race condition. Los síntomas observables:

  • "La tecla Q falla a veces."
  • "La veleta del viento no se ve" (a veces, falsa coincidencia).
  • "El humo no responde al toggle."

Todos son consecuencias en cadena del 500 de /tick:

  1. /tick falla → Godot no recibe diffs → el set de celdas activas no se actualiza.
  2. El overlay Q dispara /quadrants y AGRAVA la race → más 500.
  3. El toggle H envía /humo_virtual con éxito (es atómico, no bloquea el lock por mucho), pero como /tick no llega nunca a completarse, no se ve cambio.

La solución

Lock global en api/motor.py:

import threading

_lock = threading.Lock()

def tick() -> list[dict]:
    with _lock:
        p = _proceso_activo()
        ...

Cada función pública (ignite, tick, reset, set_viento, set_humo_virtual, pintar_firebreak, deshacer_firebreak, stats, quadrants, fuel_map, shutdown) adquiere el lock al entrar. Los helpers internos (_vivo, _arrancar, _proceso_activo) NO adquieren — sólo se llaman desde dentro de funciones públicas que ya lo tienen.

Es la cola natural de comandos: si /tick está procesando miles de CHANGEDs, /quadrants espera 200-2000 ms y sale después limpio. El cliente Godot percibe una latencia ocasional pero ningún 500.

Tests

No hay un test directo de race condition (Python en thread tras thread es difícil de reproducir determinista). Pero existe un test indirecto en api/test_endpoints.py:

def test_humo_virtual_toggle_idempotente():
    with TestClient(app) as client:
        for _ in range(10):
            client.post("/humo_virtual", json={"activo": True})
            client.post("/humo_virtual", json={"activo": False})
        # No crash, no 500

Y los tests E2E llaman /tick 15-20 veces seguidas con /snapshot intercalados. Si el lock se rompe, fallan inmediatamente.

Lección general

Cualquier vez que la API tenga estado compartido — un proceso hijo, una conexión a BBDD, un cache mutable — necesita lock aunque el framework parezca "single-threaded". FastAPI corre handlers en threadpool por defecto. La concurrencia ocurre.

Ver también Protocolo del Motor para la decisión inicial de mantener UN proceso motor con stdin/stdout, y la nota sobre alternativas (binario, GDExtension) en Integracion Godot-Motor — Recopilatorio.
