Visión general

El sistema es un pipeline de tres capas independientes. Cada capa habla con la siguiente por una interfaz mínima y bien definida. Ninguna conoce los detalles internos de las otras.

┌──────────────────────────────────────────────────────────────────────┐
│  HERRAMIENTAS (tools/)  —  OFFLINE, una vez por terreno              │
│  Convierte heightmap_16bit.png en data/world.dat (binario)           │
└──────────────────────────┬───────────────────────────────────────────┘
                           │ data/world.dat (versionado)
┌──────────────────────────▼───────────────────────────────────────────┐
│  MOTOR C++ (engine/)                                                 │
│  Proceso persistente. Carga world.dat al arrancar.                   │
│  Recibe comandos por stdin, emite CHANGED + TICK_DONE por stdout     │
└──────────────────────────┬───────────────────────────────────────────┘
                           │ stdin/stdout (texto, comandos)
┌──────────────────────────▼───────────────────────────────────────────┐
│  API (api/)                                                          │
│  Mantiene el proceso C++ vivo. Acumula CHANGED en memoria.           │
│  Expone HTTP a Godot.                                                │
└──────────────────────────┬───────────────────────────────────────────┘
                           │ HTTP / JSON
┌──────────────────────────▼───────────────────────────────────────────┐
│  GODOT (godot/)                                                      │
│  Pinta los diffs de POST /tick. No conoce nada de física.            │
└──────────────────────────────────────────────────────────────────────┘

Por qué esta separación

Godot no hace física. Solo visualiza. Si el motor cambia o se reescribe, ni la API ni Godot tienen que cambiar.

La API no hace física. Solo orquesta. Si Godot fuera reemplazado por una web app, el motor y las herramientas no cambiarían.

El motor no conoce la red. Recibe texto por stdin, devuelve texto por stdout. Se puede compilar, ejecutar y testear de forma autónoma sin levantar ningún servidor.

El terreno es estático

El DEM, los vecinos de cada cuadrante, la dirección de pendiente y, en el futuro, los coeficientes de combustibilidad por especie no cambian nunca durante la simulación. Por eso todo lo precomputable vive en data/world.dat (binario, generado por build_world) y se versiona.

El motor lo carga al arrancar y lo considera inmutable. El estado dinámico (qué celda arde) vive solo en RAM y solo en los cuadrantes activos.

Cuadrantes y activación dinámica

El terreno se trocea en una rejilla de cuadrantes:

1598 m horizontal / 50 m por cuadrante = 32 cuadrantes en X
1393 m horizontal / 50 m por cuadrante = 28 cuadrantes en Z
                                        = 896 cuadrantes totales

Cada cuadrante tiene un SubGrid de 50×50×20 = 50.000 celdas (1×1 m horizontal, ~7 m vertical). Solo los cuadrantes con actividad se procesan; el resto no existe en RAM.

INACTIVO  → no se procesa, su grid 3D no está alocado
ACTIVO    → física completa cada tick
HALO      → procesado parcialmente para propagar bordes (futuro)

Las cuatro fases de un tick

✎ Revisión 2026-06-27

El tick real ejecuta aplicarViento (#74) antes de la buoyancy y, en modo superficie (el del producto), enfriamiento dentro de aplicarReaccion (#131) e ignición por adyacencia (propagar_adyacencia, #132). El listado de abajo omite esas fases (y la sección "Compatibilidad con la física antigua" ya no es literal: reaccion.h ganó el parámetro de enfriamiento y presion/adveccion pasaron a buffers). Versión canónica en Motor de fuego — las operaciones del tick y cómo se combinan.

Cada POST /tick ejecuta sobre cada cuadrante ACTIVO los cuatro pasos físicos heredados del motor monolítico anterior, más una fuente de calor sostenido y los pasos de gestión de cuadrantes:

PARA CADA CUADRANTE ACTIVO:
  1. aplicarBuoyancy   →  el calor genera velocidad ascendente
  2. resolverPresion   →  corrige la divergencia del campo de velocidad
  3. advectar          →  transporta temperatura, densidad, reacción
  4. aplicarReaccion   →  consume combustible, FUEL→BURNING, BURNING→SMOKE
  5. sostener_burning  →  reinyecta TEMP_IGNICION en celdas BURNING

DESPUÉS:
  6. propagar_halos    →  intercambia bordes entre cuadrantes activos
  7. evaluar_desactivacion  →  desactiva cuadrantes sin actividad
  8. evaluar_activacion     →  activa vecinos cuyo borde supera UMBRAL

El orden 7→8 (desactivar antes de activar) es crítico: un vecino recién activado no se desactiva el mismo tick. Y sostener_burning es necesario porque la advección tiende a drenar el calor del foco — sin reinyección, una celda BURNING termina con temperatura cero en un par de ticks.

Protocolo motor ↔ API

El motor C++ es un proceso vivo durante toda la sesión. La comunicación es por texto, un comando o respuesta por línea:

API                              Motor C++
 │                                  │
 │── "IGNITE 275 275\n"  ──────────▶│  (no responde)
 │── "TICK\n"            ──────────▶│  (procesa)
 │                                  │
 │◀──── "CHANGED 275 4 275 BURNING 800\n" ─│
 │◀──── "CHANGED 275 5 275 AIR 250\n"      │
 │◀──── ...                                │
 │◀──── "TICK_DONE 50000\n" ───────────────│
 │── "RESET\n"  ────────────────────▶│
 │── "QUIT\n"   ────────────────────▶│  (cierre limpio)

Solo se emiten CHANGED para celdas que han cambiado de estado o temperatura más de un umbral (0.5 °C). En un tick típico viajan miles de celdas, no millones.

Ver el protocolo formal: Protocolo del Motor · Arquitectura API

Optimización: solo lo que cambia y solo lo que arde

Tres niveles de filtrado eliminan trabajo y tráfico inútil:

  1. Cuadrantes inactivos no se procesan. Tras una ignición típica solo 1–10 cuadrantes están activos de los 896 totales.
  2. El motor solo emite celdas que cambian respecto al último valor emitido por celda. La mayoría de la simulación no genera CHANGED.
  3. /tick devuelve solo los diffs del último tick (transiciones a/desde BURNING/SMOKE; los cambios de solo-temperatura se filtran). /snapshot da la foto completa para sincronizar al conectar. Es el protocolo de diffs (MVP7).

Resultado: Godot puede hacer polling a 30–60 fps sin saturar la red local.

Compatibilidad con la física antigua

Las cuatro cabeceras de física (buoyancy.h, reaccion.h, adveccion.h, presion.h) no han cambiado respecto al motor monolítico anterior. Operan sobre Grid&, que ahora es un using Grid = SubGrid; — el SubGrid (50×50×20 de un cuadrante) replica la interfaz de la antigua estructura monolítica.

Esto deja deuda técnica visible: las cabeceras siguen hablando de "Grid" en sus argumentos. La refactorización a Cuadrante& o SubGrid& directo se pospuso al cierre del rediseño porque no era necesaria para el criterio funcional. Ver: MVP — Arquitectura de Cuadrantes sección "Lo que NO entró".
