Optimización — `#66` — Precomputar divergencia fuera de Jacobi

⚠ Documento histórico (pre-refactor)

Refleja un estado anterior al refactor de caminos forestales (jun 2026); se conserva como traza del proceso. El estado actual está en Inicio.

← Optimización — Paralelismo y Coste · Optimización — 63 — Parada temprana Jacobi · Optimización — 64 — Paralelismo OpenMP

TL;DR

  • Lift the loop invariant: calcularDivergencia(g, x, y, z) se llamaba dentro del bucle de iteraciones Jacobi pese a que la velocidad NO cambia durante Jacobi (lo dice el propio comentario de presion.h). Ahora se precomputa una sola vez por llamada a resolverPresion.
  • Latencia de presión secuencial: 40.99 → 21.78 ms/tick a 100 ticks. Drop -46.9 %, criterio del issue era ≥15 %. Cumple ampliamente.
  • Bit-idéntico al baseline post-#63: BURNING/SMOKE/iters por celda exactamente iguales. test_determinismo_numerico_adveccion pasa sin tocar el snapshot. No hay cambio numérico, solo eliminación de trabajo redundante.
  • Cero regresiones: 44 API + 22+32+18 motor + 3 binarios.
  • Re-bench de #64: con la divergencia precomputada, el speedup de paralelismo a 50 k BURNING sube ligeramente (1.10× con 4 hilos vs baseline post-#63), pero el 4× sigue sin alcanzarse. El cuello de botella se ha trasladado a I/O HTTP / serialización; el cómputo ya no domina el /tick total.

Por qué esta issue se priorizó antes de #65

Originalmente el orden era #64 (paralelismo) → #65 (validación física) →

66 (memoria, opcional). Tras #64

quedó claro que el paralelismo no paga porque cada cuadrante (1.6 MB de grid) satura L3 cuando varios hilos trabajan en paralelo. El comentario de presion.h apuntaba directamente a un trabajo redundante:

"La velocidad NO cambia durante el solver Jacobi, así que la divergencia es la misma en cada iteración."

Pero calcularDivergencia(g, x, y, z) se llamaba dentro del bucle de iteraciones — recomputábamos lo mismo ~18 veces (avg iters/call tras #63). Cada recomputación gasta 6 lecturas de velocity por celda, y la velocidad está dispersa por el grid (50000 celdas × 24 bytes Vec3 ≈ 1.2 MB del grid). Levantar la invariante:

  • Reduce las operaciones de memoria por iter en ~38 % (eliminamos 6 lecturas de velocity, añadimos 1 lectura de divergencia precomputada).
  • Mejora la localidad de caché: div_buf[i] se lee linealmente durante Jacobi. Antes los reads de velocidad iban a cells contiguas que se compartían con otros campos de Celda (struct of arrays implícito pero mezclado).
  • Bit-idéntico: la divergencia es función del estado de velocidad al inicio de la llamada, idéntica en todas las iteraciones. Ya iba a serlo.

Diseño

presion.h::resolverPresion

Firma nueva:

int resolverPresion(SubGrid& g, SubGrid& buf_a, SubGrid& buf_b,
                    std::vector<float>& div_buf,
                    int iteraciones = 30, float epsilon = 0.5f);

div_buf lo provee el llamante. Tamaño mínimo: g.cells.size() (50000 floats = 200 KB). Si llega más pequeño se redimensiona — coste amortizado.

Pre-loop nuevo (un solo barrido):

std::size_t i = 0;
for (int z = 0; z < g.sizeZ; z++) {
  for (int y = 0; y < g.sizeY; y++) {
    for (int x = 0; x < g.sizeX; x++) {
      buf_a.cells[i].pressure = g.cells[i].pressure;  // warm-start
      div_buf[i] = calcularDivergencia(g, x, y, z);    // ← una vez
      ++i;
    }
  }
}

El orden (z, y, x) con x interior coincide exactamente con la indexación lineal index(x,y,z) = x + sizeX*(y + sizeY*z), así que i incrementa monotónicamente y podemos hacer cells[i] y div_buf[i] sin recalcular el índice. Es la misma micro-optimización aplicada al bucle interno de Jacobi.

Bucle Jacobi: en lugar de float divergencia = calcularDivergencia(g, x, y, z); ahora (... - div_buf[i]) / 6.0f.

simulacion.h

Nuevo miembro:

std::vector<std::vector<float>> bufs_divergencia_;

Replicado por hilo OpenMP, igual que los otros 4 buffers. Inicializado lazy en asegurar_buffers_por_hilo. En el cuerpo del bucle paralelo:

int iters = resolverPresion(c.grid,
                            bufs_presion_a_[tid],
                            bufs_presion_b_[tid],
                            bufs_divergencia_[tid]);

200 KB extra por hilo, despreciable frente a los 6.4 MB de los buffers de presión y advección.

Por qué seguimos sin llegar al 4× — análisis del cuello de botella

/tick máximo en tick 160 ≈ 10 segundos. Si miramos qué consume ese tiempo:

Fase post-#66 1-thread (160 ticks) % del /tick
Presión (motor) 134 ms/tick avg ~1.3 %
Advección (motor) 98 ms/tick avg ~1.0 %
Resto motor ~9 ms/tick ~0.1 %
I/O motor (CHANGED por stdout) resto ~30-40 %
HTTP / JSON / Python state diff resto resto

A tick 160 el motor emite 21 768 líneas CHANGED por stdout, la API las parsea, las pasa al state, calcula diffs, las serializa a JSON, las manda por HTTP. El cómputo físico es una fracción minúscula del /tick wall-clock a este tamaño.

El paralelismo del motor ya no puede mover la aguja porque el cómputo no es el dominante. Para llegar al 4× a 50 k BURNING haría falta atacar el pipeline I/O completo:

  1. Reducir CHANGED emitidos (parcialmente abordado por #59 — filtrar diffs por transición de estado renderable). A tick 160 todavía se emiten ~22k líneas/tick — mucho.
  2. Salir del protocolo texto stdin/stdout. Un protocolo binario o compartir memoria evitaría parsear 22k líneas por tick.
  3. Mover cálculo de diffs al motor C++, en lugar de en Python. La API solo recibiría los diffs ya formateados.

Estas optimizaciones están claramente fuera del scope del milestone actual. Ver "Trabajo futuro" en Optimización — Paralelismo y Coste.

Cambios

  • engine/presion.h:
  • Firma nueva con std::vector<float>& div_buf.
  • Pre-loop barre (z,y,x) rellenando buf_a.pressure (warm-start) y div_buf[i] (divergencia) simultáneamente con un único índice lineal.
  • Bucle Jacobi sustituye calcularDivergencia(...) por div_buf[i]. Comentario explicando la invariante.
  • engine/simulacion.h:
  • Nuevo bufs_divergencia_ (vector>, per-hilo).
  • Inicialización en constructor (1 hilo) y en asegurar_buffers_por_hilo.
  • tick() pasa bufs_divergencia_[tid] a resolverPresion.
