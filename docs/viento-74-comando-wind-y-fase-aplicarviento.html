TL;DR

  • Primera pieza del milestone "Viento": nuevo header engine/viento.h con aplicarViento, nueva fase en el tick() del motor que añade una aceleración horizontal uniforme al campo de velocidades, nuevo comando WIND vx vz en el protocolo, profile_.viento instrumentado.
  • La trampa de los ejes queda blindada dentro de Simulacion: el endpoint y el protocolo usan coordenadas del mundo (vx = X mundo, vz = Z mundo). El swap a la convención interna del motor (Vec3.y → Z mundo, NO Vec3.z) ocurre una sola vez en set_viento. Después aplicarViento puede sumar componente a componente sin condicionales.
  • Determinismo preservado: con viento_motor_ = (0, 0, 0) la nueva fase es no-op (sumar 0 al campo de velocidades), así que el snapshot numérico de 10 ticks pasa SIN re-snapshotar. Confirmado.
  • 19 asserts C++ + 3 bash nuevos. Test crítico: viento_Z_acumula_en_velocity_y — si alguien rompe el swap, ese test lo caza directamente.

Por qué la convención está hecha así

El milestone empezó con la documentación de la convención de ejes en docs/convenciones-ejes.md porque era bloqueante. Resumen breve:

Significado Vec3 motor API JSON Godot
Vertical velocity.z cell.y Y
Horizontal X mundo velocity.x cell.x X
Horizontal Z mundo velocity.y (¡!) cell.z Z

En el motor, Vec3.y corresponde al eje horizontal Z mundo. Esto es contraintuitivo si uno lee "vel.y" pensando que es vertical (no lo es: vel.z es el vertical, según buoyancy.h::aplicarBuoyancy).

El plan original del HANDOFF decía:

c.velocity.z += viento.z * dt * acoplamiento;   // ← BUG

Eso habría empujado humo verticalmente en lugar de lateralmente. Reescrito a:

c.velocity.y += viento.y * dt * acoplamiento;   // .y mapea a Z mundo

Y para que el resto del código no tenga que pensar en este swap, la traducción se hace una sola vez en set_viento (que recibe nombres del mundo) y se guarda en viento_motor_ con la convención motor.

Diseño

engine/viento.h (nuevo)

void aplicarViento(SubGrid& g, const Vec3& viento, float dt,
                   float acoplamiento = 0.1f) {
    const float ax = viento.x * dt * acoplamiento;
    const float ay = viento.y * dt * acoplamiento;

    for (int z = 0; z < g.sizeZ; z++) {
        for (int y = 0; y < g.sizeY; y++) {
            for (int x = 0; x < g.sizeX; x++) {
                Celda& c = g.get(x, y, z);
                if (c.state == EstadoCelda::FUEL) continue;  // sólo gases
                c.velocity.x += ax;
                c.velocity.y += ay;
                // velocity.z NO se toca: exclusivo de buoyancy.
            }
        }
    }
}

ax y ay se precalculan fuera del bucle — no cambian con la celda. Acoplamiento por defecto 0.1 (inercia moderada). Aplicación sólo a gases (AIR/BURNING/SMOKE), nunca a FUEL.

simulacion.h::set_viento

void set_viento(float vx_mundo, float vz_mundo) {
    viento_motor_.x = vx_mundo;
    viento_motor_.y = vz_mundo;   // ← swap: Z mundo va al componente .y
    viento_motor_.z = 0.0f;       // vertical: no aplica al viento
}

Único punto del código donde sucede el swap. El resto del motor lee viento_motor_ directamente y suma componente a componente.

Fase en tick()

Antes de aplicarBuoyancy en el bucle paralelo, con su propio sample de tiempo:

auto t0 = clock::now();
aplicarViento(c.grid, viento_motor_, dt);
const uint64_t ns_v = ns_desde(t0);
// ...
#pragma omp atomic update
profile_.viento += ns_v;

Orden de fases en el tick():

aplicarViento → aplicarBuoyancy → resolverPresion → advectar
              → aplicarReaccion → sostener_burning

El viento entra antes que la buoyancy porque el efecto debe estar presente cuando la presión proyecta a flujo incompresible. Si el viento fuera después de la presión, sus efectos quedarían "no balanceados" y el campo divergiría en pasos posteriores.

STATS actualizado

STATS ticks=N viento=ns buoyancy=ns presion=ns adveccion=ns reaccion=ns
      sostener=ns halos=ns desact=ns act=ns presion_iters=N presion_calls=N

viento se sitúa justo después de ticks y antes de buoyancy para reflejar el orden físico de aplicación.

Handler WIND en main.cpp

} else if (cmd == "WIND") {
    float vx = 0.0f, vz = 0.0f;
    if (iss >> vx >> vz) sim.set_viento(vx, vz);
}

Sin respuesta del motor — el comando es de set, análogo a IGNITE en ese sentido (que tampoco responde directamente). La verificación externa es ver STATS.viento > 0 tras un tick.

Decisiones tomadas durante la implementación

  1. ax y ay precalculados fuera del bucle. La aceleración por tick es constante (no depende de (x, y, z)), así que viento.x * dt * acoplamiento se calcula una vez por cuadrante en lugar de 125000 veces (50×50×50, total celdas del SubGrid).

  2. Sólo gases, no FUEL. El viento es una fuerza sobre el fluido. FUEL representa combustible sólido — un árbol no se mueve con el viento dentro del modelo. Cuando una celda FUEL ignita y pasa a BURNING, en ese momento empieza a recibir el empuje del viento.

  3. Cero comprobaciones de signo o magnitud. Confiamos en la API (issue #75) para validar vx, vz. El motor acepta lo que le manden — si Godot envía valores absurdos, el bbox del frente lo delatará visualmente.

  4. profile_.viento > 0 incluso sin viento configurado. El bucle recorre todas las celdas siempre, sumando 0 si el viento es cero. El coste es la baseline del overhead; con viento configurado debería ser comparable. No vale la pena hacer un "if (viento==0) skip" porque el overhead total es del orden de buoyancy (~1 % del tick).

  5. Posición de la fase ANTES de buoyancy. Documentado en el comentario del código. La razón: la presión proyectará el campo resultante a incompresible; si el viento entrase después de la presión, sus efectos quedarían no balanceados.

  6. Test unitario puro para aplicarViento. En lugar de probar sólo end-to-end (que mezcla todas las fases), test_aplicarViento_toca_x_e_y_no_z monta un SubGrid sintético y verifica la función aislada. Detecta bugs en la propia fase sin depender de la cadena completa.

  7. El test crítico de Z se queda corto en lo que verifica. Sólo comprueba que velocity.y se incrementa con viento (0, 10); no compara magnitudes ni signos vs un baseline. Es lo necesario y suficiente para detectar un swap roto. La validación física más profunda (bbox del frente desplazado en Z) queda para la issue #76.
