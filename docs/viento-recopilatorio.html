Hipótesis física que se demuestra

El campo de viento desplaza el frente del incendio en la dirección del viento. La magnitud del desplazamiento es proporcional a la magnitud del viento y al tiempo.

Por qué sale (casi) gratis del solver

aplicarViento sólo añade una aceleración horizontal al campo de velocidades en celdas gaseosas. El resto del solver (presión proyecta a incompresible, advección transporta los campos, buoyancy mantiene la verticalidad) hace que esa aceleración se traduzca en un desplazamiento horizontal del humo y, por reacción/convección, del frente de BURNING.

No se ha tocado advección ni presión. La feature aprovecha el solver existente como las cortafuegos.

La trampa de los ejes (resuelta)

Documentado a fondo en docs/convenciones-ejes.md. Resumen:

Significado Vec3 motor API JSON Godot
Vertical velocity.z cell.y Y
Horizontal X mundo velocity.x cell.x X
Horizontal Z mundo velocity.y (¡!) cell.z Z

El swap se hace una sola vez en Simulacion::set_viento:

void set_viento(float vx_mundo, float vz_mundo) {
    viento_motor_.x = vx_mundo;
    viento_motor_.y = vz_mundo;   // ← Z mundo va al componente .y
    viento_motor_.z = 0.0f;       // vertical: no aplica
}

A partir de ahí, aplicarViento puede sumar componente a componente sin condicionales, la API expone sólo nombres del mundo, y Godot no necesita saber nada de la convención interna.

Validado end-to-end: el test test_e2e_viento_Z_desplaza_centroide_en_cell_z falla en bloque si alguien rompe el swap.

Lectura para gestores forestales:

  • 10 m/s durante 30 ticks (5 minutos simulados) ⇒ 17 m de desplazamiento del centro del frente.
  • La asimetría del desplazamiento X vs Z (17 vs 11) es por la asimetría de fondo del solver, no del viento — el viento mismo es isótropo.
  • El frente con viento NO sólo se desplaza; también cambia su forma (n BURNING crece a +X y baja a +Z respecto al baseline). Es una observación cualitativa para una iteración futura.

Decisiones cerradas durante el milestone

  1. Documentar la convención de ejes antes de tocar código (paso 0, sesión previa). Producido docs/convenciones-ejes.md con tabla de tres componentes (motor/API/Godot), verificación empírica de cada uno, y el plan de implementación corregido para usar velocity.x/velocity.y (NO velocity.z).
  2. El swap sucede UNA sola vez en Simulacion::set_viento. El resto del motor (y la API entera, y Godot) habla en coordenadas del mundo.
  3. Sólo gases, no FUEL en aplicarViento. El viento no se aplica a combustible sólido.
  4. velocity.z (vertical) NUNCA se toca desde viento. Es exclusiva de aplicarBuoyancy.
  5. Fase aplicarViento ANTES de buoyancy, dentro del bucle paralelo por cuadrante. Razón: la presión debe proyectar el campo resultante a incompresible incluyendo el empuje del viento.
  6. API sin rango duro en vx/vz. El usuario decide la magnitud; los valores absurdos los delata el bbox del frente.
  7. DELETE/undo NO existe. Cambiar el viento es simplemente otro POST /viento. /reset también lo limpia (gracias a la línea viento_motor_ = Vec3{} en reset()).
  8. Métrica narrativa: centroide del bbox, no conteo en una región fija como en cortafuegos. La feature aquí es desplazamiento, no contención.
  9. Tests asimétricos en umbrales (5 para X, 3 para Z). Refleja la asimetría base del solver, no es un sesgo del test.

Trabajo futuro (no es de este milestone)

  1. Interacción cortafuegos × viento. Demostrar que una franja perpendicular al viento contiene; paralela al viento es sobrevolada. Sinergia narrativa fuerte para el TFG.
  2. Viento variable en el espacio: usar Cuadrante::pendiente_media y pendiente_dir (ya cargados de world.dat, ningún solver los usa todavía) para canalizar el viento por valles y subir laderas.
  3. Viento variable en el tiempo: rachas, calmas. El endpoint recibiría una serie temporal o un perfil.
  4. HUD de viento en Godot: flecha en pantalla mostrando dirección y magnitud. Requiere endpoint GET /viento que aún no existe.
  5. Optimización si hace falta: la fase aplicarViento recorre todas las celdas siempre, incluso con viento_motor_ = 0. Si alguna vez se mide y el overhead es no despreciable, añadir un if (viento.x == 0 && viento.y == 0) return; al inicio.
