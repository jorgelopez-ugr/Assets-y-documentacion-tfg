Evaluación de caminos (M4)

El núcleo científico del TFG: el fuego deja de ser una simulación y pasa a ser la función de evaluación que puntúa cada cortafuegos. Acompaña a Trazado automático A — coste y datos (M3) y Contención del cortafuegos — enfriamiento (M2).

Coste del trazado vs evaluación

Son cosas distintas (la propuesta lo subraya): - Coste del trazado (A*, M3): a priori, minimiza pendiente/longitud/biomasa/ especies para proponer el camino más barato. - Evaluación (M4): a posteriori, simula el fuego y mide si el camino funciona (contiene el incendio). La métrica final combina ambos.

Escenario de evaluación (decidido con el usuario)

  • Focos de ignición: los marca el usuario (uno o varios). Máxima flexibilidad para preguntar "si prende aquí, ¿aguanta el cortafuegos?".
  • Zona protegida = especies valiosas (pino carrasco, encina, quejigo, de especies_map.png; mismas que penaliza el A*, configurable). Es lo que el cortafuegos debe salvar.

#124 — Recolección de factores: POST /evaluar

Autocontenido y reproducible: resetea → pinta el cortafuegos → enciende los focos → corre N ticks recolectando factores. Comparar dos caminos = dos llamadas con el mismo foco.

// request
{ "focos": [{"x":799,"z":696}],
  "firebreak_cells": [[x,z], ...],   // opcional; el camino a evaluar
  "ticks": 100,                      // 1..400
  "intensidad": 1.0 }
// response (factores crudos)
{ "ticks": 100,
  "area_quemada": 1234,              // celdas que ardieron
  "protegidas_total": 50000,        // celdas de zona protegida en el mapa
  "protegidas_quemadas": 0,         // contención: 0 = perfecto
  "contenido": true,                // ¿el fuego NO llegó a zona protegida?
  "tick_primera_proteccion": null } // ticks hasta que entra (null = nunca)

Implementación: el endpoint cuenta el área quemada y las entradas en zona protegida a partir de los diffs que devuelve cada motor.tick(). La máscara de zona protegida (evaluacion.protegidas()) se cachea.

Verificado: un anillo de cortafuegos alrededor del foco reduce el área quemada; determinista; validaciones de entrada.

Métrica compuesta: POST /puntuar (rediseñada en #140)

La métrica se rediseñó a "mayor = mejor" (#140): un buen cortafuegos debe puntuar ALTO (más intuitivo). Combina 4 factores, cada uno en [0,1] con 1 = ideal, ponderados (los pesos suman 1; variarlos = análisis de sensibilidad):

factor valor [0,1] (1 = ideal) peso default
eficacia área SALVADA / baseline = (baseline − quemada)/baseline 0.35
protección fracción de zona valiosa salvada = 1 − quemadas/total 0.25
tiempo 1 si contenido; si no, cuánto TARDA el fuego en llegar (tick/ticks) 0.15
coste-eficiencia área salvada / coste del camino, normalizado suave por EFICIENCIA_MEDIA 0.25 
// request
{ "focos": [{"x":799,"z":696}],
  "firebreak_cells": [[x,z], ...],     // el camino a puntuar
  "ticks": 100,
  "pesos": {"w_eficacia":0.5, "w_proteccion":0.5, ...} }  // opcional; default arriba
// response
{ "puntuacion": 0.78,                   // mayor = mejor, [0,1]
  "desglose": { "eficacia": {"valor":.., "peso":..}, "proteccion":{...}, ... },
  "factores_crudos": { area_quemada, area_baseline, area_salvada, coste_camino, ... } }

El endpoint corre una baseline sin cortafuegos (para medir el área salvada) y la corrida con el camino, calcula el coste del camino (pathfinding.coste_camino) y puntúa (evaluacion.puntuar, función pura testeable). Comparar dos caminos = dos llamadas; gana el de puntuación más alta, y el desglose explica por qué.

La filosofía: premiar el beneficio (terreno y zona valiosa salvados, retraso del fuego) en relación al coste del camino, en vez de penalizar el daño. Un camino caro se justifica si salva mucho (factor coste-eficiencia).

Flujo en 3 fases (#134) — el bueno

/puntuar (batch) sigue existiendo para comparar dos caminos concretos, pero el flujo de uso real es en 3 fases, y la evaluación es del incendio real que ocurre (no una re-simulación):

  1. PINTAR: solo herramientas de cortafuegos (manual + auto).
  2. QUEMA: la primera chispa (ignición de siempre) cierra el pintado y abre el incendio; arranca un timer de 5 min.
  3. EVALUACIÓN automática: 5 s sin celdas ardiendo (o el timer) → POST /resultado → panel con puntuación + barra por factor + área vs baseline + contenido sí/no. Reset (R) vuelve a PINTAR.

Backend de la sesión (resultado.py): la API acumula los factores en vivo — /ignite registra focos, /firebreak el cortafuegos, /tick acumula área/ contención/tiempo (solo desde que arde). /resultado puntúa con esos factores + el coste del camino, corriendo solo una baseline sin cortafuegos (capada a 200 ticks) para normalizar el área. Así el panel refleja exactamente la partida.

Pesos fijos (equilibrado) por ahora; sliders de sensibilidad → posible extra.

Estado: M4 completo

Flujo de punta a punta: trazar (M3) → pintar/terraformar (M2) → evaluar el fuego → puntuar y comparar caminos. El fuego es ya la función de evaluación del TFG, con métrica ponderada y desglose interpretable.
