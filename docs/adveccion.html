Para qué existe este módulo

La advección responde a la pregunta: ¿cómo se mueve el material (calor, humo, combustible) a través del espacio?

Sin advección, el fuego no se propaga — solo se queda en la celda donde se encendió y se consume ahí. La advección es el mecanismo que transporta temperatura, densidad y reaction siguiendo el campo de velocidades que generan buoyancy y presión.

Las tres funciones y su relación

Son una cadena de llamadas:

advectar
  └── semiLagrangiano  (por cada celda)
        └── interpolar  (para leer un punto entre celdas)

interpolar — subrutina de geometría

static float interpolar(const Grid& g, float x, float y, float z,
                         std::function<float(const Celda&)> campo)

¿Por qué existe? El origen de una partícula en semi-Lagrangiano raramente cae exactamente en el centro de una celda — cae en algún punto flotante entre celdas. Necesitamos saber qué valor tiene el campo en ese punto continuo.

¿Cómo funciona? Rodea el punto con las 8 celdas vecinas que forman un cubo y hace un promedio ponderado: cuanto más cerca estás de un vértice, más pesa ese vértice.

punto (x=1.7, y=2.3, z=0.9)  →  fx=0.7, fy=0.3, fz=0.9

    v010 ─────── v110
   /│            /│
 v011 ─────── v111 │
  │  v000 ────│── v100
  │ /         │ /
 v001 ─────── v101

Cada vértice pesa la combinación de sus fracciones: - v000 pesa (1-fx)*(1-fy)*(1-fz) — el más lejano pesa menos - v111 pesa fx * fy * fz — el más cercano pesa más

El resultado es la interpolación trilineal estándar de la dinámica de fluidos.

✎ Es `static`

Solo se llama desde semiLagrangiano. No forma parte de la interfaz pública del módulo.

semiLagrangiano — traza de dónde viene el material

static float semiLagrangiano(const Grid& g, int x, int y, int z,
                              float dt,
                              std::function<float(const Celda&)> campo)

¿Por qué no simplemente "mover hacia adelante"? Mover partículas hacia adelante es inestable numéricamente — con dt grande el valor se dispara. La solución es hacer la pregunta al revés:

En lugar de "¿adónde va este material?", pregunta "¿de dónde viene el material que va a llegar aquí?"

destino (x, y, z)  ←  rastreamos hacia atrás
                         ox = x - dt * vel.x
                         oy = y - dt * vel.y
                         oz = z - dt * vel.z
                       interpolamos en (ox, oy, oz)

Cuando dt es negativo (paso 2 de MacCormack), el trazado va en dirección contraria — hacia adelante en el tiempo.

✎ Es `static`

Solo se llama desde advectar. No forma parte de la interfaz pública del módulo.

advectar — MacCormack en 3 pasos

Grid advectar(const Grid& g, float dt)

Función pública. Devuelve un nuevo grid con temperature, density y reaction transportados. El grid original no se modifica.

¿Por qué no basta con semi-Lagrangiano simple? El semi-Lagrangiano básico introduce difusión numérica: el fuego se vuelve borroso con cada tick porque el promedio suaviza los bordes del frente de fuego.

MacCormack añade un tercer paso que compensa ese error:

Paso 1 — adelante:    advectar el grid original con +dt
                      (estimación del siguiente estado)

Paso 2 — atrás:       advectar el resultado del paso 1 con -dt
                      (estimar cuánto error introdujo el paso 1)

Paso 3 — corrección:  resultado = adelante + 0.5 * (original - atrás)
                      (compensar el error)

Es el mismo principio que el método predictor-corrector en ecuaciones diferenciales ordinarias.

Los tres grids temporales

Grid adelante  = g;   // resultado del paso 1
Grid atras     = g;   // resultado del paso 2
Grid resultado = g;   // salida final (paso 3)

Esto implica tener 3 grids en memoria simultáneamente durante la advección. Con el grid actual (100×10×100), el pico de memoria es ~10 MB. Ver Cuadrante y Celda.

Qué se advecta y qué no

Campo ¿Se advecta? Motivo
temperature ✅ El calor se mueve con el fluido
density ✅ El humo se mueve con el fluido
reaction ✅ El combustible se transporta
velocity ❌ No se auto-advecta (ver nota abajo)
pressure ❌ Campo auxiliar, se recalcula cada tick

Nota: lo que falta (MVP futuro)

La auto-advección de velocidad (u·∇u) no está implementada. En el paper de Nvidia, la velocidad también se transporta a sí misma, lo que produce movimientos de fluido más realistas (vórtices, turbulencia).

Actualmente la velocidad solo se modifica por buoyancy y presión, no por transporte. Es una simplificación consciente para MVP 3 — el motor funciona correctamente sin ella.

Residuo histórico

El comentario de la función advectar dice:

// Transporta temperatura y densidad a través del campo de velocidades.

Esto era correcto antes de que se añadiera reaction como campo advectado. El comentario no está actualizado — en realidad transporta los tres campos.
