El subsistema de logging quedó en desuso. La carpeta logs/ (incluidos rotate_logs.py y su test) se sacó del control de versiones el 2026-06-15: ya no se versiona nada de logs/. Esta nota se conserva solo como referencia histórica; no describe el estado actual del proyecto.

⚠ Doc parcialmente histórico (rediseño "Arquitectura de Cuadrantes")

El motor C++ del rediseño no expone --debug ni escribe engine.log. La sección sobre el debug del motor C++ y la captura desde api/motor.py ya no aplica. Lo que sigue vigente:

  • api/logger.py (módulo centralizado) — sin cambios.
  • logs/api.log con la actividad HTTP de la API — sigue funcionando.
  • logs/rotate_logs.py — sigue siendo válido (rota cualquier .log).

El motor nuevo emite CHANGED y TICK_DONE por stdout y se comunica con la API por texto: si necesitas trazar lo que ve la API en un tick, añadirías log.debug(...) en api/motor.py directamente.

Por qué existe

Cuando algo falla (tick que no responde, proceso C++ muerto, estado corrupto) no había rastro de qué ocurrió ni cuándo. El módulo api/logger.py añade observabilidad de la capa Python sin tocar la lógica de simulación.

Estructura de ficheros

logs/
├── api.log          ← actividad de la API (requests, ticks, errores)
├── rotate_logs.py   ← script de rotación manual
└── .gitkeep         ← garantiza que la carpeta se versiona vacía

Los .log están en .gitignore — solo se versiona la carpeta.

api/logger.py — el módulo centralizado

Cómo usarlo en cualquier módulo de la API

from logger import get_logger

log = get_logger(__name__)
log.info("mensaje informativo")
log.debug("detalle solo visible en DEBUG_MODE=1")
log.error("algo salió mal")

get_logger configura los handlers una sola vez (flag _configurado) aunque se llame desde varios módulos. El root logger de Python se configura globalmente.

Handlers

Handler Cuándo activo Fichero Nivel
RotatingFileHandler Siempre logs/api.log INFO / DEBUG según DEBUG_MODE
StreamHandler (consola) Solo si DEBUG_MODE=1 stdout DEBUG

La rotación del handler de fichero es automática a 5 MB, guardando 3 backups:

RotatingFileHandler(API_LOG_FILE, maxBytes=5*1024*1024, backupCount=3)

Esto es diferente a la rotación manual de rotate_logs.py. El handler rota dentro de la sesión activa; el script rota y comprime entre sesiones.

Variable de entorno DEBUG_MODE

# API silenciosa — solo logs a fichero, nivel INFO
uvicorn main:app --app-dir api

# API verbosa — fichero + consola, nivel DEBUG, motor en modo debug
DEBUG_MODE=1 uvicorn main:app --app-dir api

DEBUG_MODE afecta a dos cosas a la vez: 1. Logger de la API — activa nivel DEBUG y el handler de consola 2. Motor C++ — lanza el proceso con --debug y captura su stderr en logs/engine.log

Qué se loguea en cada endpoint

Endpoint Nivel Qué se loguea
arranque API INFO ruta de initial_state.json, número de celdas
POST /reset INFO confirmación + celdas cargadas
POST /ignite INFO coordenadas (x, z), y si la celda es nueva o existente
POST /tick DEBUG celdas BURNING/SMOKE tras el tick
GET /snapshot DEBUG foto completa para sync inicial
POST /firebreak INFO celdas de cortafuegos pintadas
errores ERROR mensaje de excepción original

El endpoint de alta frecuencia (/tick) usa nivel DEBUG para no saturar el log en producción.

logs/engine.log — debug del motor C++

Qué escribe el motor

Con --debug, el motor C++ escribe una línea a stderr tras procesar cada tick:

DEBUG tick=1 burning=1 smoke=0 temp_max=799.97
DEBUG tick=2 burning=3 smoke=1 temp_max=743.21
DEBUG tick=3 burning=7 smoke=2 temp_max=698.05

Campos: - tick — número de tick acumulado desde que arrancó el proceso - burning — celdas en estado BURNING en ese instante - smoke — celdas en estado SMOKE - temp_max — temperatura máxima del grid completo (°C)

Por qué stderr y no stdout

El protocolo entre la API y el motor usa stdout para el JSON de respuesta. Mezclar debug en stdout rompería la deserialización. stderr está libre para mensajes humanos sin interferir con el protocolo.

Cómo la API captura el stderr

En api/motor.py, cuando DEBUG_MODE=1:

# Lanzamiento con stderr=PIPE
_proceso = subprocess.Popen(
    cmd + ["--debug"],
    stdin=subprocess.PIPE,
    stdout=subprocess.PIPE,
    stderr=subprocess.PIPE,
    text=True,
)

# Tras cada tick, leer la línea de debug
linea_debug = proceso.stderr.readline().strip()
if linea_debug:
    _get_engine_logger().debug(linea_debug)

La lectura de stderr es síncrona y funciona porque el motor escribe exactamente una línea por tick. No hay riesgo de deadlock porque stdout ya fue leído antes.

El engine logger (logs/engine.log) tiene su propio RotatingFileHandler con propagate = False — no contamina el log de la API.

Ejemplo de sesión con DEBUG_MODE=1

DEBUG_MODE=1 uvicorn main:app --app-dir api
