MVP 2 — Proceso Persistente y Terreno

⚠ Documento histórico (pre-refactor)

Refleja un estado anterior al refactor de caminos forestales (jun 2026); se conserva como traza del proceso. El estado actual está en Inicio.

← Inicio · Arquitectura del Sistema · Arquitectura API

Objetivo

Eliminar el overhead de spawn de proceso por tick y conectar el motor con el heightmap real del terreno, preparando el grid para la visualización en Godot.

Decisiones arquitectónicas

  • Protocolo línea a línea — un JSON compacto por línea en stdin → una línea en stdout. Requirió cambiar serializar.h de dump(2) a dump().
  • Proceso singleton — _get_proceso() en motor.py relanza el proceso si muere.
  • Grid 100×10×100 — balance velocidad/fidelidad. ~16×14 píxeles del PNG por celda XZ, ~15m verticales por nivel Y.
  • PNG no versionado — heightmap_16bit.png está en .gitignore. El script es el artefacto; el PNG lo aporta el usuario.

// ANTES — moría tras un tick
std::string entrada(std::istreambuf_iterator<char>(std::cin), {});

return 0;

// DESPUÉS — vive toda la sesión
while (std::getline(std::cin, linea)) {
    Grid g = stringAGrid(linea);
    std::cout << gridAString(tick(g, p)) << "\n";
    std::cout.flush();
}

#18 — api/motor.py con proceso persistente

Método Latencia media
subprocess.run (un proceso por tick) ~1.4 ms/tick
Popen persistente ~0.3 ms/tick
Mejora 5× más rápido

Con el grid real 100×10×100: ~63ms/tick (~15 ticks/segundo). El cuello de botella es el solver Jacobi, no el proceso.
