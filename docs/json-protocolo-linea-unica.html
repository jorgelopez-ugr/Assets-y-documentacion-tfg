Peculiaridad: el motor exige JSON compacto en una sola línea — heredocs multilínea causan fallos silenciosos

← Inicio · Protocolo del Motor · MVP 4 — Propagación real de fuego

⚠ Documento histórico — protocolo JSON eliminado

El protocolo JSON del motor ya no existe. Fue sustituido en el rediseño "Arquitectura de Cuadrantes" por un protocolo de comandos texto (Protocolo del Motor). El bug concreto descrito aquí ya no es reproducible — el nuevo motor lee comandos como IGNITE, TICK, RESET, QUIT línea a línea.

La lección general sigue siendo válida para cualquier protocolo stdin/stdout línea a línea: si construyes el comando en bash con un heredoc multilínea, std::getline solo lee la primera línea y el proceso falla silenciosamente. Ahora aplicado al nuevo protocolo: printf "IGNITE 275 275\nTICK\nQUIT\n" ✓; cat << EOF ... EOF ✗ si el contenido tiene saltos no intencionados.

MVP afectado: MVP 4 (motor monolítico, ya retirado) Archivos involucrados (históricos): engine/main.cpp (antiguo) · engine/test_propagacion.sh (eliminado)

El protocolo de comunicación del motor

El motor C++ es un proceso persistente que lee JSONs de stdin y escribe JSONs en stdout. El protocolo es línea a línea:

stdin  → un JSON compacto + "\n"  (una línea)
stdout ← un JSON compacto + "\n"  (una línea de respuesta)

main.cpp usa std::getline para leer:

while (std::getline(std::cin, linea)) {
    if (linea.empty()) continue;
    Grid estadoActual = stringAGrid(linea);  // parsea una línea completa
    ...
}

La consecuencia: si el JSON tiene saltos de línea, getline lee solo la primera línea, que es JSON inválido. El motor lo descarta silenciosamente (la línea vacía se salta con continue) o intenta parsear un fragmento incompleto y el comportamiento es indefinido.

Cómo se manifestó el bug

Al escribir engine/test_propagacion.sh, el estado inicial se definió con un heredoc multilínea para legibilidad:

# INCORRECTO — produce JSON multilínea
ESTADO_INICIAL=$(cat << 'JSON'
{"grid_size":{"x":7,"y":5,"z":7},"cells":[
  {"x":3,"y":2,"z":3, ...},
  {"x":2,"y":2,"z":3, ...}
]}
JSON
)

El motor recibía la primera línea ({"grid_size":{"x":7,"y":5,"z":7},"cells":[) que no es JSON válido. En lugar de dar error visible, producía salida vacía en stdout. El script interpretaba la salida vacía como un fallo del motor y moría en el primer tick.

El síntoma era confuso: la compilación era exitosa, el motor arrancaba, pero el script se interrumpía silenciosamente sin mensaje de error claro.

La corrección

El JSON debe ir en una sola línea. En bash:

# CORRECTO — JSON compacto en una variable de una sola línea
ESTADO_INICIAL='{"grid_size":{"x":7,"y":5,"z":7},"cells":[{"x":3,"y":2,"z":3,...},{"x":2,"y":2,"z":3,...}]}'

Si el JSON es largo, generarlo desde Python (que produce una sola línea con json.dumps) es la opción más robusta:

ESTADO_INICIAL=$(python3 -c "import json; print(json.dumps({...}))")

La API (api/motor.py) ya hace esto correctamente:

proceso.stdin.write(json.dumps(estado) + "\n")

json.dumps por defecto produce JSON compacto en una sola línea. El "\n" es el terminador que el motor espera.

Regla para cualquier script que use el motor directamente

  1. Todo JSON enviado al motor debe ser una sola línea (sin saltos).
  2. Generar JSONs desde Python con json.dumps() garantiza esto automáticamente.
  3. En bash, usar variables de cadena directa (sin heredoc) o generar con python3 -c.
  4. Si el motor no responde, verificar primero si el JSON tiene saltos de línea — es el fallo silencioso más común al trabajar con el motor desde scripts.
