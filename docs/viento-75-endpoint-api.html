Viento — `#75` — Endpoint `POST /viento`

TL;DR

  • Segunda pieza del milestone "Viento": expone por HTTP el comando WIND del motor introducido en #74.
  • POST /viento con payload {vx: float, vz: float} en coordenadas del mundo (X mundo, Z mundo, m/s). Sin rango impuesto — el usuario decide la magnitud.
  • El swap a la convención interna del motor sigue blindado en C++: la API ni siquiera ve Vec3.y. Godot recibe una superficie limpia con los nombres vx/vz.
  • No toca el motor: sólo capa HTTP + wrapper. La API pública existente (/tick, /snapshot, /ignite, /reset, /firebreak) no cambia su contrato.
  • 9 tests nuevos (3 wrapper + 6 endpoint). Cero regresiones.

Diseño

api/motor.py — wrapper nuevo

def set_viento(vx: float, vz: float) -> None:
    """Fire-and-forget: el motor no responde.

    vx y vz son las componentes en coordenadas del MUNDO. El swap a
    la convención interna del motor (Vec3.y ↔ Z mundo) sucede dentro
    de Simulacion::set_viento en C++.
    """
    p = _proceso_activo()
    p.stdin.write(f"WIND {vx} {vz}\n")
    p.stdin.flush()

Patrón idéntico a ignite — escribe una línea y vuelve. No lee respuesta porque el comando WIND del protocolo no emite nada (al contrario de FIREBREAK_BATCH que sí emite FIREBREAK_DONE N).

api/main.py — modelo + endpoint

class Viento(BaseModel):
    vx: float
    vz: float

@app.post("/viento")
def viento(payload: Viento):
    motor.set_viento(payload.vx, payload.vz)
    return {"status": "ok"}

Sin validación de rango: vx y vz son float libres. Si el usuario envía 1000 m/s, el solver lo aceptará; el resultado visual lo delatará (frente catapultado fuera del mapa). Esta es una decisión consciente — no queremos imponer un límite artificial que pueda no ser apropiado para todos los escenarios.

_CAMPOS_STATS actualizado

motor.py mantiene una tupla de campos esperados en la línea STATS. Añadimos viento ahí (en la posición correcta, justo después de ticks) para que motor.stats() recoja el campo y /stats lo exponga sin cambios al endpoint.

Decisiones tomadas durante la implementación

  1. Sin rango duro en vx/vz. Pydantic acepta cualquier float. El usuario asume responsabilidad por valores absurdos — y el bbox del frente delatará el problema visualmente. Imponer un rango arbitrario (p.ej. ±50 m/s) sería inventar un límite sin justificación física.

  2. No expongo el getter viento_motor() del motor. Existe para tests C++; la API no lo necesita. Si Godot quiere saber el viento actual (para mostrarlo en HUD), se añadirá un endpoint GET /viento cuando haga falta. Por ahora es un push de Godot al motor.

  3. /stats recoge automáticamente el campo viento. La cabecera del motor ya incluye viento=ns; el parser de motor.parse_stats pasa cualquier campo conocido (en la tupla _CAMPOS_STATS), así que basta con añadir "viento" a la tupla — no hay endpoint nuevo, no hay cambio en /stats por código.

  4. Fire-and-forget igual que ignite. El motor no responde a WIND. Esto es coherente con un comando de set (no de consulta). La verificación externa pasa por STATS.viento > 0 tras un tick.

  5. No verifico la magnitud del swap desde la API. Los tests de la convención (viento_X y viento_Z en C++) ya cubren ese comportamiento. Desde la API basta con verificar que el endpoint recibe y propaga al motor; verificar que el motor empuja en el eje correcto es responsabilidad de los tests C++ (donde se observa velocity.x/y/z directamente).
