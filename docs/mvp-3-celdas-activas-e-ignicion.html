MVP 3 — Celdas Activas e Ignición

⚠ Documento histórico (pre-refactor)

Refleja un estado anterior al refactor de caminos forestales (jun 2026); se conserva como traza del proceso. El estado actual está en Inicio.

← Inicio · Arquitectura del Sistema · Arquitectura API

Objetivo

Conectar el motor con Godot para visualizar fuego propagándose sobre el terreno real. El bucle funciona de extremo a extremo; la visualización puede ser simple (esferas).

Decisiones arquitectónicas

  • GET /active_cells en lugar de polling de GET /state. El estado completo de 100.000 celdas serializado pesa varios MB. Las celdas activas son siempre un subconjunto pequeño — diferencia de varios órdenes de magnitud.

  • Ignición en el script, no en la API — el punto de fuego inicial se inyecta al generar el JSON del terreno (terrain_to_json.py --ignition-x X --ignition-z Z) y se carga con POST /reset. La API no necesita saber nada de ignición.

  • Y de ignición = superficie del heightmap — la celda de fuego se coloca al nivel que le corresponde según el heightmap, no a Y=0 fijo.

Issues cerradas

#21 — GET /active_cells

Filtra celdas con state == "BURNING" o state == "SMOKE". Devuelve {"cells": [{x, y, z, state}]}. Sin estado inicializado → {"cells": []} (sin HTTP 400, para no romper el polling).

#22 — Flags --ignition-x/z en terrain_to_json.py

  • Sin flags → cells: [] (comportamiento anterior intacto)
  • Con ambos flags → una celda BURNING en (ignition_x, y_surface, ignition_z) con temperature: 800.0, reaction: 1.0
python3 terrain_to_json.py ../heightmap_16bit.png --ignition-x 50 --ignition-z 50

Correspondencia grid → Godot

pos_x = (x / 100.0) * ancho_del_planemesh
pos_z = (z / 100.0) * alto_del_planemesh
pos_y = altura_terreno_en(x, z) + offset_visual

El eje Y del motor (0–9) no se mapea directamente a la altura real de Godot — las esferas pueden flotar a altura fija sobre el punto XZ del terreno.
