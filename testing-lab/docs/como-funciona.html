Cómo está construido el testing-lab

La idea en una frase: probar y dibujar las funciones reales del motor, una a una, sin arrancar el motor. Ni API, ni Godot, ni bucle de simulación, ni red. Aquí está el truco que lo hace posible.

El truco: el motor es header-only

El motor de física vive entero en cabeceras (engine/buoyancy.h, presion.h, adveccion.h…): la definición completa de cada función está en el .h, no solo su firma. Eso permite que una pequeña sonda C++ haga #include de esa cabecera y llame a la función exactamente la misma que compila el motor — sin enlazar nada del resto (ni main.cpp, ni la clase Simulacion, ni la API).

Incluir una cabecera es leerla, no tocarla. La sonda #include-a el código del motor en modo solo lectura: no lo modifica, no lo reescribe, no lo reimplementa. Por eso lo que medimos es el comportamiento real del motor, no una copia que podría desviarse.

El recorrido de una medición

Cada experimento sigue siempre la misma tubería. El C++ ejecuta la función; Python solo lanza el binario y dibuja lo que sale por la salida estándar:

engine/op.h
la función real del motor
→
probe_op.cpp
#include + llama a la función sobre datos sintéticos
→
bin/probe_op
binario (g++ −O2) que vuelca CSV a stdout
→
op.py
lanza el binario, parsea el CSV y dibuja
→
out/op.png · gif
la figura

En paralelo, un test_op.py lanza la misma sonda y verifica un invariante físico del CSV (p. ej. "la divergencia baja", "el pico difunde"), para que las figuras nunca mientan.

Qué se ejecuta de verdad… y qué no

SÍ se ejecuta

  • La función real del motor (aplicarBuoyancy, resolverPresion, advectar…), compilada con el mismo g++ −O2.
  • Sobre un SubGrid 50×50×20 de verdad (la misma estructura de datos del motor).
  • Con datos sintéticos que yo controlo: cualquier campo de temperatura, velocidad o estado que quiera.

NO se ejecuta

  • El bucle de simulación (clase Simulacion): ticks, cuadrantes, activación, halos.
  • La API FastAPI ni el protocolo stdin/stdout del motor.
  • Godot, la red, world.dat, el DEM real.
  • Nada de física en Python: solo lanza el binario y dibuja.
Por qué importa Como aislamos la función del resto, podemos meterle cualquier carga — mil cuadrantes, un disco girando mil veces, un foco imposible — sin el límite que impone la simulación real. Eso es justo lo que permite medir el techo de rendimiento (la curva O(n) de buoyancy) y ver la física paso a paso (Jacobi iteración a iteración) que en el motor completo quedaría enterrada.

El ejemplo más pequeño posible

Así de corta es una sonda. Construye un grid, pone un dato, llama a la función real y vuelca el resultado:

// probe (C++)
#include "../../engine/buoyancy.h"   // la función REAL del motor (solo lectura)
#include <cstdio>

int main() {
    SubGrid g; g.inicializar();
    Celda& c = g.get(25, 25, 0);
    c.temperature = 800;                 // dato sintético que YO controlo
    c.state = EstadoCelda::BURNING;

    aplicarBuoyancy(g);                  // ← la MISMA función que llama el motor

    std::printf("%f\n", c.velocity.z);   // vuelco el resultado a stdout
}

Y el experimento Python no sabe nada de física: lanza el binario y dibuja.

# experiment (Python)
salida = subprocess.run(["bin/probe_buoyancy"], capture_output=True, text=True).stdout
vz = float(salida)                       # parsea el CSV
plt.plot(...); plt.savefig("out/buoyancy.png")   # Python SOLO dibuja

Anatomía de cada operación

Por cada función del motor hay cuatro piezas, siempre con el mismo patrón:

PiezaLenguajeHace
probes/probe_<op>.cppC++llama a la función real sobre un grid sintético, vuelca CSV
experiments/<op>.pyPythonlanza la sonda, parsea, grafica (PNG/GIF) en out/
tests/test_<op>.pyPythonverifica un invariante físico del CSV
docs/<op>.htmlHTMLexplica qué muestra cada figura y qué demuestra

La invariante de seguridad

El laboratorio nunca modifica el motor ni nada del proyecto: solo lo usa. Todo lo que genera (binarios, CSV, PNG, GIF, el entorno Python) vive bajo testing-lab/ y está en .gitignore. Tras correr el lab, git status solo debería mostrar archivos de testing-lab/ — nunca nada bajo engine/, api/, godotUI/, data/ ni tools/.

En resumen: el motor pone la física (en cabeceras header-only), C++ la ejecuta aislada sobre datos a medida, y Python solo orquesta y dibuja. Probamos el código real del motor sin el peso del motor entero. Vuelve al índice de operaciones.
