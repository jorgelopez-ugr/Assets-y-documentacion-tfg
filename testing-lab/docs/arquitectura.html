La API como orquestador

Tres procesos que no se conocen entre sí —Godot, FastAPI y el motor C++— y una capa en medio que los coordina. Aquí está quién hace qué, por qué están separados y cómo viaja de verdad una petición de punta a punta.

El sistema no es un programa: son tres piezas independientes hablando por dos fronteras distintas. Godot pinta y recoge clics; el motor C++ calcula física; y entre ambos, la API de FastAPI hace de director de orquesta: recibe HTTP, traduce, arranca y serializa el motor, compone operaciones de alto nivel y devuelve JSON. Ninguna de las tres piezas sabe hacer el trabajo de las otras — y eso es a propósito.

La regla de oro de la separación: el motor C++ no sabe nada de red · la API no sabe nada de física · Godot no sabe nada de simulación. Si una capa empieza a hacer el trabajo de otra, la frontera se rompe — y con ella la posibilidad de probar cada pieza por separado (es justo lo que permite el testing-lab).

Por qué la API es un orquestador y no un proxy

Un proxy se limitaría a reenviar. La API hace tres trabajos que ninguna otra capa puede hacer:

TrabajoQué resuelveDónde vive
Traducir HTTP/JSON del exterior ⇄ protocolo de líneas stdin/stdout del motor. Godot habla JSON; el motor habla TICK\n. motor.py
Gestionar el proceso Arrancar el binario C++ una vez (lazy), mantenerlo vivo entre comandos, serializar accesos concurrentes y cerrarlo limpio al apagar. motor.py · lifespan
Componer Encadenar primitivas (reset → cortafuegos → ignición → N ticks) en operaciones de alto nivel como /evaluar y /puntuar. main.py · evaluacion.py

El motor es un proceso persistente, no una llamada

Un error tentador sería lanzar el motor C++ en cada petición. No: el binario se arranca una sola vez con subprocess.Popen y se mantiene vivo, porque cargar world.dat (el DEM completo) es caro y el estado del incendio debe persistir entre ticks.

El arranque es perezoso: la API levanta sin tocar el motor («el motor C++ se lanzará al primer comando», dice el log de arranque). Las consultas de solo lectura que llegan antes —/status, /stats, /quadrants— responden con valores por defecto en vez de forzar el arranque, para no contaminar el ciclo de vida.

Un solo proceso, muchos hilos: el lock

FastAPI sirve peticiones en varios hilos, pero el motor es un único proceso con un único stdin y un único stdout. Si dos endpoints escriben y leen a la vez, los comandos y sus respuestas se entrelazan y el parser explota leyendo basura:

ValueError: could not convert string to float: '6.211.3827' dos floats de líneas CHANGED distintas, concatenados por dos hilos pisándose

La defensa es un threading.Lock global en motor.py: cada función pública (tick, ignite, pintar_firebreak…) adquiere el lock al entrar, de modo que un comando completo —escribir y leer su respuesta entera— es atómico frente a los demás. Serializa el motor sin que Godot ni la API tengan que coordinarse.

En código, ese recorrido es exactamente este bucle de lectura: la API escribe el comando y se queda leyendo líneas hasta ver el centinela TICK_DONE, acumulando los CHANGED por el camino:

# motor.py — tick()  (dentro de  with _lock:)
p.stdin.write("TICK\n"); p.stdin.flush()
cambios = []
while True:
    linea = p.stdout.readline()
    if linea.startswith("TICK_DONE"):     # centinela de fin
        return cambios
    if linea.startswith("CHANGED"):       # una celda que cambió
        _, wx, wy, wz, st, t = linea.split()
        cambios.append({"x": int(wx), "y": int(wy), "z": int(wz),
                        "state": st, "temperature": float(t)})

Antes de devolver el JSON, la API pasa esos cambios por state.py, que filtra: de todos los CHANGED del motor solo reenvía los que alteran algo renderable (apariciones, desapariciones y transiciones BURNING↔SMOKE); un cambio de temperatura que no cambia el estado visible se descarta. Así Godot recibe diffs mínimos, no el volcado entero del motor.

Lo que demuestra el recorrido La API nunca calcula el tick: lo pide, traduce el protocolo de líneas a JSON, recorta lo que el cliente no necesita y mantiene el proceso C++ ordenado bajo el lock. El motor, por su lado, ni se entera de que existe una red. Cada capa hace una sola cosa.

La frontera de abajo: el protocolo de líneas

Entre la API y el motor no hay JSON ni HTTP: hay líneas de texto por stdin/stdout. Un comando por línea; las respuestas terminan en un centinela (TICK_DONE, FIREBREAK_DONE, MODE…) que la API espera antes de soltar el lock.

RESET(silencioso)limpia el estado dinámico del incendio

Los comandos que el motor no responde (IGNITE, WIND, RESET, SMOKE_VIRTUAL) son fire-and-forget: la API escribe y sigue, sin bloquearse a leer. Solo los que devuelven datos (TICK, FIREBREAK_BATCH, QUADRANTS…) obligan a esperar el centinela.

Orquestación de alto nivel: el núcleo científico

Aquí la API deja de ser un traductor y se vuelve un compositor. Evaluar la calidad de un cortafuegos no es una primitiva del motor: es una secuencia de primitivas que la API encadena en un escenario limpio y reproducible —_simular_escenario:

RESET
estado del incendio a cero
→
FIREBREAK

→
IGNITE ×focos

→
N × TICK

→

Sobre esa secuencia se construyen las operaciones del TFG:

  • POST /evaluar — corre un escenario y devuelve los factores crudos del incendio (área quemada, si el fuego cruzó el cortafuegos, cuándo alcanzó zonas protegidas).
  • POST /puntuar — corre dos escenarios: uno con el camino y una baseline sin cortafuegos («no hacer nada»), para medir el área salvada, y combina todo en una métrica compuesta ponderada. Variar los pesos = análisis de sensibilidad.
  • POST /trazar — el A* calcula el camino de menor coste y devuelve el mismo vector de celdas que produciría el dibujado manual; el cliente lo manda a /firebreak. El algoritmo reusa la tubería del usuario, no una paralela.
Por qué importa Comparar dos caminos forestales es comparar dos llamadas a /puntuar con el mismo foco y los mismos pesos. Esa reproducibilidad —reset limpio, misma semilla de escenario, factores normalizados contra la baseline— es lo que hace que la evaluación sea ciencia y no anécdota. Y vive entera en la capa de orquestación: el motor solo sabe arder.

Una frontera, una unidad: metros reales

Toda la conversación Godot↔API↔motor ocurre en metros reales del terreno (X ∈ 0–1598, Z ∈ 0–1393), nunca en índices de grid. Las conversiones internas —por ejemplo el swap de la convención de ejes del viento (Vec3.y ↔ Z mundo)— suceden dentro del motor C++, no en la frontera. La API y Godot hablan siempre el idioma del mundo físico; el motor traduce a su sistema interno puertas adentro.

En una frase: Godot quiere, el motor calcula, y la API decide cuándo, en qué orden y traduciendo qué. El valor del orquestador no está en hacer física ni en pintar, sino en mantener las otras dos capas ignorantes la una de la otra — y en componer sus primitivas en la evaluación de caminos, que es el núcleo del TFG.
